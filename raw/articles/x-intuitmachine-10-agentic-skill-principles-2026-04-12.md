## 核心洞察：为什么有些人获得 100x 结果

### 错误假设

> 当人们发现 AI 系统可以戏剧性更有效 — 不是 2 倍好，是 10 倍或 100 倍 — 他们假设是因为更聪明的模型。更好的算法。更多参数。AI 内部的秘密配方。

### 现实

> **不是。获得非凡结果的人使用和其他人相同的模型。区别在于他们如何包装这些模型 — 具体说，如何编写 skills。**

---

## 什么是 Skill

> **Skill 是可复用文档，教 AI 如何做某事。** 不是在一个特定情况下做什么，而是如何 approach 整个任务类别。

**比喻**:
- **Skill = 食谱**
- **Order = 单次订单**

| 对比项 | Order（订单） | Skill（技能） |
|--------|--------------|--------------|
| **本质** | "告诉我 X 是否发生" | "这是调查 X 的方法" |
| **复用性** | 一次，一种情况 | 数百种不同用例 |
| **具体性** | 一切具体 baked in | 描述过程，参数提供世界 |
| **比喻** | "告诉厨师今晚做什么菜" | "给厨师一本食谱" |

---

### 案例：Order vs Skill

**Order（不是 Skill）**:
```
"Analyze the customer feedback and summarize the key themes."
分析客户反馈并总结关键主题。
```
- 工作一次，一种情况
- 然后消失
- 无法复用

**Skill（真正的技能）**:
```
Skill: Thematic Analysis

Parameters:
- CORPUS (the documents)
- QUESTION (what you're looking for)
- DEPTH (quick scan or deep dive)

Process:
1. Read through the corpus noting initial impressions
2. Identify recurring patterns — what comes up again and again?
3. Name the themes you've found
4. For each theme, pull representative examples
5. Assess: which themes matter most for the QUESTION?
6. Write a synthesis organized by importance
```

**关键差异**:
- Skill 描述**过程** — 无论分析客户反馈/员工调查/学术论文/法律证词都有效的方法
- **Skill 是食谱。参数是食材。**

> **为什么重要**: 一个设计良好的 skill 可处理数百不同用例。不用每次写新指令，用不同参数调用相同 skill。建一次食谱，永远做不同餐。

---

## 原则 1：食谱非订单

**核心**: Skill 应该是可复用过程，非单次指令。

**测试**:
- 能否用这个 skill 论证相反结论？
- 如果能用"调查举报人"skill 调查"这人被封口了吗？"和"这投诉是捏造的吗？" — 并基于证据获得适当不同答案 — 那是真正的 skill
- 如果总是指向一个方向无论证据 — 是伪装的 prompt

---

## 原则 2：教如何思考，非得出结论

**错误写法**:
```
"Step 4: Conclude that the evidence supports concern about patient safety."
步骤 4：得出结论证据支持对患者安全的担忧。
```

**问题**:
- 预先决定结果
- 把 AI 变成批准你结论的傀儡
- 即使你对，skill 在证据指向不同时无用

**正确写法**:
```
"Step 4: Weigh the evidence for and against the hypothesis. Consider:
- Does the timeline make sense?
- Are there alternative explanations?
- What evidence would change your mind?
Reach your own conclusion based on what you find."

步骤 4：权衡支持和反对假设的证据。考虑：
- 时间线合理吗？
- 有替代解释吗？
- 什么证据会改变你的想法？
基于发现得出自己的结论。
```

**关键差异**:
- 第一版**替换判断**
- 第二版**调用判断**
- AI 必须真正思考问题 — 这是用 AI 的整个意义

---

## 原则 3：判断 vs 计算分离

**每个 AI 系统步骤属于两类之一，混淆它们是最常见设计错误**:

| 类别 | 判断 (Judgment) | 计算 (Computation) |
|------|----------------|-------------------|
| **定义** | 智能所在的地方 | 可靠性所在的地方 |
| **例子** | 读文档决定什么重要 | 数据库查询 |
| | 权衡竞争考虑 | 算术 |
| | 识别看似不同情况是相同模式 | 排序列表 |
| | 感觉某事不对劲即使说不清为什么 | 数字数 |
| | | 检查日期是否在范围内 |
| **AI 表现** | ✅ 真正擅长 | ❌ 不可靠 |
| **传统软件** | ❌ 无法做 | ✅ 完美做 |

---

### 直觉测试：8 人 vs 800 人座位安排

**问 AI 安排 8 人晚餐座位**，考虑谁和谁合得来/谁不应坐一起：
- 会表现出色 — 权衡个性/感知社交动态/做判断呼叫
- 这是**判断问题** — 把所有人记在心里感觉出好安排

**问 AI 安排 800 人同样约束**:
- 产生看起来像座位表但不可见违反约束的东西
- 自信把 Alice 放 Bob 旁边即使他们在排除列表
- 输出看起来合理但安静错误
- 这是**优化问题** — 需要算法，非直觉
- AI 没有算法，所以幻觉一个

**原则**:
> **Skills 应明确编排这个边界。标记哪些步骤需要判断（让 AI 思考）和哪些需要计算（调用工具）。不要让 AI 做算术。不要让计算器做解释。**

## 原则 4:Diarization 能力（综合综合）

> **这是 AI 能做而数据库查询不能的**: 读五十份关于一个人的文档，注意其间矛盾，追踪故事如何随时间变化，写一页 profile 捕捉这人实际是谁。

**我们称之为 Diarization** — 将分散信息合成为结构化智能。

---

### 案例：会议组织者理解创始人

| 来源 | 内容 |
|------|------|
| **数据库** | 公司名和行业 |
| **申请表** | "AI 基础设施" |
| **与顾问 1:1** | 主要担心计费和成本归属 |
| **GitHub commits** | 最近工作 80% 在支付模块 |

**Diarization Skill 输出**:
```
SAYS: "Datadog for AI agents"
ACTUALLY BUILDING: FinOps tool disguised as observability
GAP: Positioning doesn't match execution. Either pivoting or unclear on identity.

说："AI 代理的 Datadog"
实际构建：伪装成可观察性的 FinOps 工具
差距：定位与执行不匹配。要么转型要么身份不清。
```

**关键洞察**:
> **没有关键词搜索找到这个。没有嵌入相似度浮现这个。AI 必须读一切，把矛盾记在心里，对实际发生什么做判断呼叫。**

**原则**:
> **构建读一切并综合的 skills。不要预过滤到"相关"文档 — 在读之前不知道什么相关。力量来自看到完整画面后出现的综合。**

## 原则 5:Resolver 模式（路由系统）

### 错误：20000 行指令

> 我为 AI 助手写了 20000 行指令。每个怪癖，每个教训，每个遇到的模式。我以为更多指导会有帮助。

> **让事情更糟。AI 注意力有限。用信息淹没它意味着错过重要的。埋在 20000 行中，当前任务的关键指令变得不可见。**

---

### 正确：200 行指针

> **修复：200 行。但那 200 行不是指令 — 它们是指针。**

```
当你在做 X，加载文档 Y。
当你看到模式 A，咨询 skill B。
```

**这是 Resolver** — 在正确时刻加载正确上下文的路由系统。

| 场景 | Resolver 动作 |
|------|--------------|
| 开发者修改 prompt | 在发布前加载评估指南 |
| 分析师问关于客户 | 加载该客户历史 |
| 看到特定模式 | 加载对应 skill |

> **Resolver 知道哪个知识何时重要。**

**比喻**:
> 伟大助手不会在你问问题时给你读整个公司手册。他们知道哪页对这个重要问题并正好打开到那。Resolver 是那助手。

**原则**:
> **不要 everywhere 加载 everything。构建 sensing 需要什么并正好加载那个的 resolvers。你 AI 的注意力宝贵 — 花在当下任务重要的事上。**

## 原则 6：三层架构

```
┌─────────────────────────────────────┐
│  Skills Layer (顶层)                 │
│  - Rich documents                  │
│  - Process, judgment, wisdom       │
│  - Natural language                │
│  - 90% value lives here            │
│  - Edit to improve system          │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  Harness Layer (中层)                │
│  - Thin (~200 lines of code)       │
│  - Runs AI in loop                 │
│  - Manages context                 │
│  - Calls tools                     │
│  - No business logic               │
│  - No judgment calls               │
│  - Just plumbing                   │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  Tools Layer (底层)                  │
│  - Fast, simple programs           │
│  - One thing reliably              │
│  - Query database / Read file      │
│  - Same input → same output        │
└─────────────────────────────────────┘
```

**原则**:
> **将智能推入 skills。将执行推入 tools。保持 harness 薄。**

**为什么**:
| 层级 | 当 AI 改进时 |
|------|-------------|
| **Skills** | 自动变得更好（判断在 skills 里） |
| **Harness** | 如果嵌入智能 = 无法治理的混乱 |
| **Tools** | 保持完美可靠（只是代码） |

**反模式**:
- ❌ "Fat harness"带 40 个工具定义吃掉上下文
- ❌ 业务逻辑嵌入编排
- ❌ Skills 是薄的后思考
- **这完全颠倒**

## 原则 7：窄/快/笨工具

### 通用工具陷阱

| 问题 | 说明 |
|------|------|
| **慢** | 通用浏览器自动化可能 15 秒截图 - 点击 - 等待 - 读<br>专用工具正好做那工作流 100 毫秒 → **150x 更快** |
| **膨胀上下文** | 40 个工具定义在开始前吃掉 AI 注意力预算 |
| **隐藏复杂性** | 当工具试图"聪明"，把判断埋在某处看不见无法改进 |

---

### 正确：窄/快/笨

| 特征 | 说明 |
|------|------|
| **一件事** | 每个工具做一件事 |
| **快** | 半秒内完成 |
| **笨** | 不解释或决定 — 只执行 |

> **解放洞察**: 软件不再必须珍贵。如果需要正好做 X 的工具，建正好那工具。可能花 30 分钟。AI 甚至可帮助写它。不再需要时删除它。**工具是脚手架，非架构。**

---

## 原则 8：研究 OK 响应

### 三种用户反应

| 反应 | 含义 | 应该关注吗 |
|------|------|-----------|
| **Great** | "这正是我需要的！" | — |
| **OK** | "这…还行，我想。" | ✅ **改进住在这里** |
| **Bad** | "这完全错误。" | ❌ 是 bugs 非改进机会 |

---

### 为什么 OK 重要

> **"Bad" 响应通常是明显失败。** 系统崩溃。工具超时。skill 完全错任务。这些是 bugs，非精炼机会。

> **"OK" 响应意味着系统几乎工作。** 机制起作用。应用判断。但某事不足。profile 准确但错过关键洞察。推荐合理但不完全对。综合完整但感觉薄。

> **这个差距 — OK 和 Great 之间 — 是 skills 可改进的地方。**

---

### 改进方法

```
1. 读 AI 产生什么
2. 读用户说什么
3. 问：差距是什么？
4. 修改 skill 关闭差距
```

**真实案例**:
- 活动匹配 skill 获得 12% "OK" 评级
- 团队读反馈，识别模式 — 创始人按声明行业匹配但实际工作在不同行业
- 添加规则："当有人说 AI 基础设施但代码主要在计费，分类为 FinTech"
- **OK 评级降到 4%**

> **那规则来自检查"还行"和"伟大"之间的差距。**

---

## 原则 9：无一次性工作

### 纪律

> **"不允许做一次性工作。如果我要求你做某事且是可能再次发生的事，你必须：**
>
> 1. **第一次手动做几个例子**
> 2. **给我看输出**
> 3. **如果我批准，编码成 skill"**

**测试简单**: **如果必须问某事两次，你失败了。**

---

### 为什么重要

| 对比项 | Clever Prompt | Skill |
|--------|--------------|-------|
| **保存** | ❌ 忘记 | ✅ 保存，版本化，永远可用 |
| **运行时间** | 仅当你运行 | 凌晨 3 点你睡觉时运行 |
| **容量** | 一次 | 处理一千实例不累 |
| **模型改进** | 无自动改进 | **自动变得更好** |

> **每个写的 skill 是系统永久升级。**

---

### 复合效应

> **想象有 100 skills 的系统，每个处理某类工作。每个 skill 可靠运行。每个 skill 在模型改进时改进。随时间，系统积累能力像公司积累资产。但不像物理资产，skills 不折旧。它们复合。**

**心态转变**:
> **停止视 AI 交互为对话。开始视它们为构建永久能力的机会。** 每次解决问题，问：这能成为 skill 吗？

## 原则 10：参数化技能

### 案例：/match Skill

**Skill 定义**: 配对或分组实体按某些标准

**参数**:
- ENTITIES: 匹配谁？
- CRITERIA: 什么构成好匹配？
- CONSTRAINTS: 必须遵守什么规则？

---

### 同一 Skill，三完全不同能力

| 场景 | ENTITIES | CRITERIA | CONSTRAINTS | 输出 |
|------|----------|----------|-------------|------|
| **会议分组讨论室** | 1200 创始人 | 行业相似度 | 每室 30 人 | 创始人按行业分组 |
| **意外午餐** | 600 创始人 | 跨行业新奇 | 每桌 8 人，无见过面的 | 创始人与惊喜他们的人分组 |
| **现场社交** | 现在楼里任何人 | 最近邻嵌入 | 1:1 配对，排除最近会议 | 任何人走进即时匹配 |

> **同一 skill。同七步。完全不同行为。**

**洞察**:
> **设计良好的 skill 是带参数的方法。参数提供具体世界 — 数据/标准/约束。Skill 提供过程 — 判断/步骤/智慧。设计 skill 一次，然后在甚至没想象的上下文中永远调用它。**

> **这是将 AI 从好奇心转为力量倍增器的杠杆。一个 skill，一百用例。设计食谱；做不同餐。**

## 终极原则：编码化

> **这十个原则指向单一纪律：编码化。**

| 遇到 | 编码化 |
|------|--------|
| 重复任务 | 成 skill |
| 有效判断 | 如何做 |
| 模式 | 系统可识别的模式 |

> **获得 100x 结果的人不更聪明。不用秘密模型。只是无情地将工作编码成 skills — 然后让那些 skills 大规模运行，随时间改进，复合。**

> **一年后，你的 skills 将在比今天更好的模型上运行。每个现在写的 skill 是对那未来能力的投资。每个编码的判断变成永不失去的杠杆。**

> **系统复合。建一次。永远运行。**

> **写食谱。做饭。分享食谱。移到下一个。**

> **那是实践。**

## 对 KilroyContentBot 的启示

### 可借鉴点
1. **Skill vs Order 区别** — 建可复用过程非单次指令
2. **判断与计算分离** — AI 做判断，工具做计算
3. **Diarization 能力** — 读一切综合，注意矛盾
4. **Resolver 模式** — 动态加载相关上下文
5. **三层架构** — Skills(厚)/Harness(薄)/Tools(快)
6. **窄工具设计** — 一件事/<0.5 秒/无解释
7. **OK 响应学习** — 关注"还行"和"伟大"差距
8. **永久技能化** — 问两次=失败，编码成 skill
9. **参数化技能** — 同 skill 不同参数不同行为
10. **编码化纪律** — 100x 结果来自编码化非更聪明模型

### 可实施
- 将内容创作流程编码为 Skills
- 分离判断步骤（选题/角度）和计算步骤（字数统计）
- 实施 Diarization 读取所有来源后综合
- 创建 Resolver 动态加载相关记忆/模板
- 重构为三层架构
- 构建窄快工具（字数统计/格式检查）
- 追踪 OK 响应并改进 skills
- 将重复任务编码为永久 skills
- 设计参数化 skills 支持多场景
