## 🎯 核心问题

**大多数人的 LLM+ 文档体验是 RAG 模式：**

```
上传文件集合 → LLM 在查询时检索相关块 → 生成答案
```

**问题：**
- ❌ LLM 每次问题都从头重新发现知识
- ❌ 没有积累
- ❌ 问需要综合 5 个文档的微妙问题，LLM 必须找到并拼凑相关片段
- ❌ 没有构建起来的东西

**RAG 系统（NotebookLM、ChatGPT 文件上传等）都这样工作。**

---

## 💡 核心想法

**不是仅在查询时从原始文档检索，LLM 增量构建并维护一个持久的 wiki — 一个结构化的、相互链接的 markdown 文件集合，位于你和原始源之间。**

### 工作流程

```
添加新源
  ↓
LLM 读取它
  ↓
提取关键信息
  ↓
整合到现有 wiki
  ↓
更新实体页面
  ↓
修订主题摘要
  ↓
注意新数据与旧主张的矛盾
  ↓
加强或挑战 evolving 综合
```

**关键区别：**
- ✅ 知识编译一次，然后保持最新
- ✅ 不在每次查询时重新推导
- ✅ wiki 是持久、复合的产物
- ✅ 交叉引用已经在那里
- ✅ 矛盾已经被标记
- ✅ 综合已经反映你读过的一切
- ✅ wiki 随着每个源和每个问题变得更丰富

## 🤖 分工

| 角色 | 人类负责 | LLM 负责 |
|------|----------|----------|
| ** sourcing** | 源的探索、选择 | 无 |
| **探索** | 方向引导 | 无 |
| **提问** | 问正确的问题 | 无 |
| **总结** | 无 | 所有总结工作 |
| **交叉引用** | 无 | 所有交叉引用 |
| **归档** | 无 | 所有归档 |
| **簿记** | 无 | 所有簿记 |

**实践设置：**
```
LLM 代理（一侧） + Obsidian（另一侧）
  ↓
LLM 根据对话编辑
  ↓
你实时浏览结果
  ↓
跟随链接、检查图视图、阅读更新页面
```

**Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。**

## 📋 应用场景

### 1️⃣ 个人

追踪自己的目标、健康、心理学、自我提升：
- 归档日记条目
- 文章、播客笔记
- 随时间建立结构化自我画像

---

### 2️⃣ 研究

数周或数月深入研究一个主题：
- 阅读论文、文章、报告
- 增量构建综合 wiki
- evolving 论文

---

### 3️⃣ 读书

逐章归档：
- 为人物、主题、情节线索建立页面
- 它们如何连接
- 结束时拥有丰富的伴侣 wiki

**类比：** 粉丝 wiki（如 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page)）— 数千个相互链接页面覆盖人物、地点、事件、语言，由志愿者社区多年构建。你可以个人构建类似的东西，LLM 做所有交叉引用和维护。

---

### 4️⃣ 商业/团队

LLM 维护的内部 wiki：
- 输入：Slack 线程、会议转录、项目文档、客户通话
- 可能有人类审查更新
- wiki 保持最新因为 LLM 做没人想做的维护工作

---

### 5️⃣ 其他场景

| 场景 | 说明 |
|------|------|
| **竞争分析** | 积累竞争对手知识 |
| **尽职调查** | 结构化调查记录 |
| **旅行计划** | 组织旅行信息 |
| **课程笔记** | 综合课程材料 |
| **爱好深挖** | 任何随时间积累知识的场景 |

## 🏗️ 三层架构

### 1️⃣ 原始源

**你的策划源文档集合：**
- 文章、论文、图像、数据文件
- **不可变** — LLM 从它们读取但永不修改
- **真相源**

---

### 2️⃣ Wiki

**LLM 生成的 markdown 文件目录：**
- 摘要、实体页面、概念页面、比较、概述、综合
- **LLM 完全拥有这一层**
- 创建页面、更新、维护交叉引用、保持一致性
- **你阅读它；LLM 编写它**

---

### 3️⃣ Schema

**文档（如 CLAUDE.md 或 AGENTS.md）告诉 LLM：**
- wiki 如何结构化
- 约定是什么
- 摄入源、回答问题、维护 wiki 时遵循什么工作流

**这是关键配置文件 — 它让 LLM 成为纪律严明的 wiki 维护者，而非通用聊天机器人。**

**你和 LLM 随时间共同进化它。**

---

## 🔄 三个核心工作流

### 1️⃣ 摄入 (Ingest)

**流程：**
```
 dropping 新源到原始集合
  ↓
告诉 LLM 处理它
  ↓
LLM 读取源
  ↓
与你讨论关键要点
  ↓
在 wiki 中写摘要页面
  ↓
更新索引
  ↓
更新 wiki 中相关实体和概念页面
  ↓
追加条目到日志
```

**单个源可能触及 10-15 个 wiki 页面。**

**偏好：**
- ✅ 一次摄入一个源，保持参与
- ✅ 你阅读摘要、检查更新、指导 LLM 强调什么
- ✅ 也可以批量摄入多个源，较少监督

**关键：** 开发适合你风格的工作流，并在 schema 中记录供未来会话使用。

---

### 2️⃣ 查询 (Query)

**流程：**
```
你问问题
  ↓
LLM 搜索相关页面
  ↓
阅读它们
  ↓
综合答案 + 引用
```

**答案形式（取决于问题）：**
- markdown 页面
- 比较表
- 幻灯片（Marp）
- 图表（matplotlib）
- canvas

**关键洞察：**
> 好答案可以归档回 wiki 作为新页面。你要求的比较、分析、发现的连接 — 这些有价值，不应消失在聊天历史中。这样你的探索像摄入源一样在知识库中复合。

---

### 3️⃣ 检查 (Lint)

**定期让 LLM 健康检查 wiki：**

**查找：**
- 页面间矛盾
- 新源已取代的过时主张
- 无入链的孤立页面
- 被提及但缺乏自己页面的重要概念
- 缺失的交叉引用
- 可用网络搜索填充的数据空白

**LLM 擅长：**
- 建议新调查问题
- 建议新源查找

**这保持 wiki 随增长健康。**

---

## 📄 两个特殊文件

### index.md（内容导向）

**wiki 内容目录：**
- 每个页面列出 + 链接 + 一行摘要
- 可选元数据（日期、源计数）
- 按类别组织（实体、概念、源等）
- **LLM 每次摄入更新它**
- 查询时 LLM 先读索引找到相关页面，然后深入

**效果：** 在中等规模（~100 源、~数百页面）下效果惊人地好，避免基于嵌入的 RAG 基础设施需求。

---

### log.md（时间顺序）

**附加式记录发生了什么及何时：**
- 摄入、查询、lint 通过
- **技巧：** 每个条目以一致前缀开始（如 `## [2026-04-02] ingest | Article Title`）
- 日志可用简单 unix 工具解析：
  ```bash
  grep "^## \[" log.md | tail -5  # 最后 5 个条目
  ```
- 给你 wiki 演进时间线
- 帮助 LLM 理解最近做了什么

## 🛠️ 工具建议

### 搜索引擎

**当 wiki 增长时需要：**
- 小规模：index 文件足够
- 大规模：需要proper搜索

**推荐：** [qmd](https://github.com/tobi/qmd)
- 本地 markdown 搜索引擎
- 混合 BM25/向量搜索 + LLM 重排序
- 全部本地设备
- CLI（LLM 可 shell 调用）+ MCP 服务器（LLM 可作为原生工具使用）

**也可自己构建简单版本 — LLM 可帮助 vibe-code 朴素搜索脚本。**

---

### Obsidian 插件

| 插件 | 用途 |
|------|------|
| **Web Clipper** | 浏览器扩展，将网页文章转为 markdown |
| **Marp** | markdown 幻灯片格式，生成演示 |
| **Dataview** | 在页面 frontmatter 上运行查询，生成动态表格 |
| **Graph View** | 最佳方式查看 wiki 形状（什么连接什么、哪些是枢纽、哪些是孤立） |

---

### 图像下载技巧

**Obsidian 设置：**
1. 设置 → Files and links → "Attachment folder path" 设为固定目录（如 `raw/assets/`）
2. 设置 → Hotkeys → 搜索 "Download" → "Download attachments for current file" 绑定到热键（如 `Ctrl+Shift+D`）
3. 剪贴文章后按热键，所有图像下载到本地

**好处：**
- ✅ LLM 可直接查看和引用图像
- ✅ 不依赖可能失效的 URL

**注意：** LLM 无法原生一步读取带内联图像的 markdown — 变通方法是让 LLM 先读文本，然后单独查看一些或所有引用图像获取额外上下文。有点笨拙但足够好用。

## 💡 核心洞察

### 洞察 1：维护负担

**维护知识库的乏味部分不是阅读或思考 — 是簿记。**

**人类放弃 wiki 因为：**
```
维护负担增长快于价值
```

**LLM 不无聊、不忘更新交叉引用、可一次触及 15 个文件。**

**结果：**
```
wiki 保持维护因为维护成本接近零
```

---

### 洞察 2：分工

| 人类 | LLM |
|------|-----|
| 策划源 | 总结 |
| 指导分析 | 交叉引用 |
| 问好问题 | 归档 |
| 思考意义 | 簿记 |
| | 维护一致性 |

---

### 洞察 3：Vannevar Bush 的 Memex（1945）

**精神相关：**
- 个人、策划知识存储
- 文档间关联轨迹
- Bush 的愿景比web更接近这个：私有、积极策划、文档间连接与文档本身一样有价值
- **他无法解决的部分：谁做维护**
- **LLM 处理那个**

## 📝 实施指南

### 目录结构建议

```
wiki/
├── raw/              # 原始源（不可变）
│   ├── articles/
│   ├── papers/
│   └── assets/       # 图像
├── wiki/             # LLM 生成的 wiki
│   ├── entities/
│   ├── concepts/
│   ├── summaries/
│   └── comparisons/
├── index.md          # 内容目录
├── log.md            # 时间顺序日志
└── CLAUDE.md         # Schema/约定
```

---

### Schema 约定建议

**在 CLAUDE.md 或 AGENTS.md 中定义：**

```markdown
# Wiki 结构
- 实体页面格式
- 概念页面格式
- 摘要页面格式
- 交叉引用约定

# 工作流
- 摄入工作流（步骤）
- 查询工作流（步骤）
- Lint 工作流（步骤）

# 约定
- 页面命名
- 链接格式
- 元数据格式（YAML frontmatter）
- 日志条目格式
```

---

### 页面格式示例

**实体页面：**
```markdown
# 实体名称

## 摘要
一句话摘要。

## 关键事实
- 事实 1
- 事实 2

## 相关概念
- [[概念 1]]
- [[概念 2]]

## 来源
- [[源 1]]
- [[源 2]]

## 历史
- [2026-04-02] 初始创建
- [2026-04-05] 更新事实 3
```
