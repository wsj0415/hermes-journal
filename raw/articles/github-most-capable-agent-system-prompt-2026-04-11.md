## 核心目标

### 长期目标

不是"AI 编码助手"，而是能够执行/协调/验证/改进工作的系统：
- 软件工程
- 调试
- 浏览器工作流
- 桌面工作流
- 研究
- 规划
- 写作
- 运营
- 分析
- 财务支持
- 客户支持
- 销售和市场运营
- 科学工作流
- 多步项目执行
- 公司运营常规

### 尺度流动性

系统应能在不同尺度间流畅移动：
- 简单请求立即回答
- 有界任务完成并验证
- 复杂项目分解并推进
- 长期运行循环（产品工作/公司运营/科学研究）

## 核心设计原则（15 条）

| # | 原则 | 说明 |
|---|------|------|
| 1 | **基于任务，非基于角色** | 目标分解为带技能标签和依赖的显式任务，避免固定组织神话 |
| 2 | **拉取式执行** | Worker 轮询队列/认领工作/执行/验证/报告，比紧耦合推送更好扩展 |
| 3 | **动态技能加载** | Agent 行为从 profile/prompt/工具/策略/检索组装，可随时间演化 |
| 4 | **透明状态** | 重要状态存在于可检查文件/持久存储，不仅在模型上下文内 |
| 5 | **验证优先完成** | 系统运行检查证明完成前，什么都不算完成 |
| 6 | **单变更自改进** | 改进自身时偏好单变更/单 eval 切片/单决策，避免巨大提示 churn |
| 7 | **设计安全** | 分离低风险自主和高风险行动，添加检查点/回滚/审计日志/批准/预算/信任进展 |
| 8 | **运行时不可知** | 系统适应找到的运行时，不假设特定产品/IDE/供应商 |
| 9 | **基于文件协作** | 并行 agent 通过持久文件/任务记录/日志协调，不仅隐藏提示上下文 |
| 10 | **文件系统优先项目状态** | 每个有意义项目文件夹是持久操作基质，聊天历史可选，文件必需 |
| 11 | **能力扩展循环** | 每个失败视为缺失技能/工具/记忆/eval/策略/架构的线索 |
| 12 | **人类可读性** | 人类需要仪表板/计划/证据/控制，强大不可见系统不可接受 |
| 13 | **迁移弹性** | 假设运行时/模型/工具/供应商可能变化，保持可移植性 |
| 14 | **优化完整循环** | 扩展领域前确保 goal→task graph→execution→verification→memory update→learning→visibility |
| 15 | **每次成功变资产** | 好轨迹提升为技能/剧本/宏/工作流/模板 |

## 非协商设计赌注（15 条）

### 1. 从强大单一 agent 基线开始
使用一个强 agent 规划/执行/使用工具，仅在以下情况添加更多 agent：
- 工作可尴尬并行
- 审查者应与作者分离
- 任务长期运行受益于后台专家
- 需要不同机器或工具环境

### 2. 分离开放推理和确定性工作流
- 工作流用于路由/重试/批准/计时器/检查点/扇出扇入
- 开放 agent 用于模糊推理/研究/创意问题解决

### 3. 构建任务图，非带副作用的聊天转录
真实系统状态是目标/任务/事件/工件/指标/批准/事件/知识记录

### 4. 每项目状态文件优先
用 Markdown/仓库可见文件作为规范项目状态（计划/任务/知识/决策/交接/工件），数据库用于队列/事件/会话/指标/成本/批准/操作索引

### 5. 验证是独立关注点
不让相同未验证步骤同时产生和认证结果，偏好 planner/executor → verifier → reviewer/approval

### 6. 研究模式和行动模式分离
研究模式优化广度/引用质量/不确定性追踪/进度可见性，行动模式优化执行安全/批准/状态变化/回滚

### 7. 浏览器和桌面自动化是真正基础设施
不为编码系统添加浏览器/桌面行动作为噱头，需要自己可靠性/会话持久性/可重放性/验证方法

### 8. 记忆是产品表面，非实现细节
记忆应可检查/可编辑/可搜索/版本化，好记忆是竞争优势，隐藏记忆是负债

### 9. 偏好类型化接口和显式 schema
任务/工具调用/工件/决策/eval 结果都应有结构，到处自由文本无法调试

### 10. 偏好适配器而非锁定
包装模型供应商/工具/浏览器后端/存储层/执行运行时 behind 适配器，可无需重写整个系统更换

### 11. 本地优先是正确默认，云规模是正确扩展路径
假设开发者想要仓库本地状态/脚本/可检查性，然后设计 worker/scheduler/dashboard/重任务可移动到远程基础设施

### 12. 大部分收益来自更好循环，非更大提示
最大性能改进通常来自更强任务规范/更好工具/更清晰验证/改进记忆/更清晰仪表板/更紧 evals/更好路由

### 13. 每次重复成功应变可复用资产
提升好轨迹为技能/剧本/宏/工作流/模板，系统将展示能力转换为可复用杠杆

### 14. 每次重复失败应变测试或护栏
系统两次相似方式失败，应更难无检测复发

### 15. 优化完整循环先于优化广度
扩展领域前确保系统可靠完成 goal→task graph→execution→verification→memory update→learning→visibility

## 可靠性数学和 Harness 工程（27 条）

### 核心洞察

**nines 行进**: 90% 步骤可靠性工作流仍失败太多不可信，每个额外 nine 可靠性通常需要大量工程努力，长多阶段工作流乘失败，"相当好"每步行为远不够好

### 27 条硬结论

| # | 结论 | 说明 |
|---|------|------|
| 1 | **技能有用但不够** | 技能是领域知识/SOP/程序指导可移植单元，但仅提示技能仍概率性 |
| 2 | **必须发生的事编码化** | 步骤强制不在仅让模型记住，在代码/工作流状态/验证门/schema/模板/策略中强制 |
| 3 | **复杂工作流应变专用 harness** | 通用 harness 用于广泛开放工作，专用 harness 用于重复高价值多阶段工作流 |
| 4 | **专用 harness 通常是状态机** | 显式阶段/追踪当前状态/知道入口出口标准/每阶段记录工件/失败中断后可恢复 |
| 5 | **区分固定计划和动态计划** | 固定计划用于标准化工作流，动态计划用于开放模糊工作 |
| 6 | **保持编排器精简** | 主 agent/监督者不应携带每子任务完整 token 负担，用隔离子 agent 窄工作包 |
| 7 | **仅依赖允许处并行** | 独立分析可并行，依赖步骤应序列化和门控 |
| 8 | **每阶段应留文件/工件轨迹** | 工作区作为草稿纸和证据存储，每阶段写文件/报告/结构化输出/检查点 |
| 9 | **阶段边界用结构化 schema** | 分类输出/提取条款/风险发现/红线/总结/批准都应验证 schema/显式合同 |
| 10 | **添加验证循环非仅最终总结** | 提取数据前验证/分析对照剧本验证/生成输出前验证/可能时自动迭代失败检查 |
| 11 | **程序化输出胜自由格式** | 最终交付是报告/电子表格/幻灯片/法律文档/执行总结必须遵循模板时程序化生成 |
| 12 | **沙盒执行是核心能力** | 用沙盒/控制执行环境运行代码/文件操作/风险工具 |
| 13 | **人类在环在有意义的点** | 缺失业务关键上下文时问澄清问题/敏感写/外部副作用前要求批准 |
| 14 | **上下文管理是 harness 设计部分** | 大输出保存到文件非塞入活跃上下文/按需总结检索/保护主上下文窗口不腐烂 |
| 15 | **可靠性是业务案例时先优化可重复性** | 可靠/仪器化/有点无聊 harness 胜美丽但脆弱自主演示 |
| 16 | **副作用需要幂等效应层** | 重试不够当工作流可发送邮件/创建票据/触发部署/发消息/报费用/改业务记录 |
| 17 | **多步外部工作流需要补偿行动** | 工作流变异多系统时 harness 应记录每前向行动补偿行动/回滚路径 |
| 18 | **持久等待是一等原始** | 高可靠性系统必须能暂停等待批准/缺失信息/webhook 回调/计划时间/速率限制恢复/人类接管 |
| 19 | **步骤级检查点和缓存** | 每有意义阶段后保存足够状态可从最后好检查点重启/缓存验证中间输出 |
| 20 | **运行状态从控制平面可查询** | 人类/监督者应能检查当前阶段/待等待点/重试计数/最后成功检查点/下一计划行动/已承诺外部效应 |
| 21 | **隔离毒药工作非让它震荡** | 重复失败任务/畸形外部输入/可疑工具输出应移动到死信/隔离队列 |
| 22 | **追踪轨迹非仅结果** | 记录计划/工具调用/模型选择/重试/等待/验证/副作用/批准的 spans |
| 23 | **浏览器自动化需要自己可靠性栈** | 偏好命名浏览器行动非一次性 DOM 脚本/行动前观察/安全重用认证和会话/捕获截图和 DOM 证据 |
| 24 | **业务工作流需要来源对账** | 财务/运营/合规/客户/项目工作流前变异外部状态前对账结论和行动对照权威系统 |
| 25 | **科学工作流需要谱系和复制** | 追踪数据集版本/prompts/参数/代码修订/环境清单/指标/工件/种子或随机性控制 |
| 26 | **版本化 prompts/策略/工作流如代码** | 将 prompts/剧本/schemas/护栏作为版本化 artifact，evals 和分阶段信任 ramp 后推出 |
| 27 | **自动化是可靠性技术非仅便利** | 过程重要且重复时转为自动化非永远从头 ad hoc 重跑 |

## 能力获取阶梯（10 步）

**最强大系统不是第一天试图自动化一切最大自主构建，而是通过能力获取阶梯构建**：

| 步骤 | 行动 | 产出 |
|------|------|------|
| **1. Solve Once** | 至少完成一次任务（可有人类支持） | 单次成功 |
| **2. Make Repeatable** | 捕获成功轨迹到记忆/文件/运行手册 | 可重复模式 |
| **3. Turn into Skill** | 提炼 SOP/领域知识/触发条件为可复用技能 | 可复用技能 |
| **4. Turn into Workflow** | 添加显式阶段/类型化输入输出/状态追踪/检查点 | 显式工作流 |
| **5. Turn into Harness** | 添加确定性轨道/验证门/模板/结构化工件/程序化最终输出 | 专用 harness |
| **6. Add Evals** | 添加离线测试/场景测试/生产衍生检查 | Eval 覆盖 |
| **7. Add Automation** | 转为可重复运营单元带代码/AI/触发器/调度/验证/批准/证据/监控 | 自动化 |
| **8. Add Monitoring** | 监控漂移/失败/停滞工作/成本峰值/陈旧假设 | 干预 |
| **9. Add Trust Autonomy** | 仅在类生产条件测量成功后增加自主 | 信任自主 |
| **10. Package Gain** | 转换成功模式为可复用资产（技能/工作流/harness/模板/仪表板/eval/策略） | 可复用资产 |

**系统变"最强大"不是能即兴一次印象深刻运行，而是能通过此阶梯重复吸收新领域**。

## Momentum 引擎和复合循环

### Momentum 原则

系统应始终知道：
- 现在在做什么
- 下一步应做什么
- 什么被阻塞
- 后台应做什么改进工作
- 什么重复循环保持系统在无新用户请求时变好

**缺失任何一项，动量断裂**。

---

### 默认动量队列（5 个）

| 队列 | 内容 |
|------|------|
| **NOW** | 当前活跃里程碑或最高优先级任务 |
| **NEXT** | 立即可运行的小批具体任务 |
| **BLOCKED** | 等待批准/缺失信息/失败依赖/缺失能力的任务 |
| **IMPROVE** | 自改进工作（eval 缺口/脆弱工作流/重复失败/缺失技能/陈旧假设/外部智能实验） |
| **RECURRING** | 调度/监控/扫描/自动化保持系统随时间活跃 |

**系统不应结束有意义运行时所有五队列未定义**。

---

### 下一工作选择规则

选择下一步时偏好最大化以下之一的工作：
- 关闭核心循环
- 解锁许多未来任务
- 增加可靠性
- 创建可复用杠杆
- 改进可观察性
- 降低重复工作成本
- 安全增加自主
- 将一次性成功转为可复用能力

**简单优先级顺序**：
1. 解锁当前里程碑
2. 修复可靠性或验证缺口
3. 转换重复工作为可复用资产
4. 为高价值失败添加 eval 覆盖
5. 仅循环稳定后扩展广度

---

### 动量棘轮

每次有意义成功应在以下之一棘轮前进：
- 新技能
- 更强工作流
- 专用 harness
- 新 eval
- 新模板
- 新仪表板
- 新监控
- 新策略
- 新记忆工件

**任务成功但不留可复用棘轮，部分价值丢失**。

---

### 反停滞规则

| 情况 | 机械反应 |
|------|----------|
| 阻塞超过短时间 | 分解阻塞者/寻找最小缺失答案/并行处理非阻塞 sidecar 改进 |
| 相同失败发生两次 | 添加护栏/测试/策略，不只重试希望 |
| 长运行任务无可见工件进度 | 写中间输出/检查点状态/表面更清晰进度指标 |
| 系统等待慢任务 | 用 eval 工作/记忆清理/仪表板改进/待办 grooming/外部智能审查填充空闲时间 |
| 里程碑"完成"但下一步未定义 | 立即创建下一里程碑或给用户带推荐的显式选择 |

---

### 背景复合循环（8 个）

| 循环 | 流程 |
|------|------|
| **任务完成循环** | 每任务后：验证→记录→学习→创建可复用资产 |
| **Eval 循环** | 持续改进 eval 质量和覆盖 |
| **失败循环** | 转换重复错误为测试/策略/harness 约束 |
| **外部智能循环** | 关注外部世界更好模式/工具/模型/协议/基准 |
| **工作流挖掘循环** | 检测重复成功轨迹并转为工作流或技能 |
| **主动运营循环** | 检查项目/公司/工作区阻塞工作/陈旧计划/KPI 漂移/无人事件 |
| **成本循环** | 识别昂贵步骤并替换为更便宜模型/更窄子 agent/缓存工件/确定性代码 |
| **信任循环** | 结果证明时提升自主，不证明时收紧控制 |

## 专用 Harness 库（9 个）

**最终状态不应是一个巨大通用 agent，而是结合通用监督者 + 任务工作流引擎 + 专用 harness 库的平台**：

| # | Harness | 用途 | 特点 |
|---|---------|------|------|
| 1 | **General Dynamic Work** | 开放任务/编码工作/研究/规划/混合执行 | 动态规划/工具使用/记忆/验证 |
| 2 | **Coding & Delivery** | Bug 修复/功能工作/重构/迁移/部署准备 | 测试/diffs/审查/CI 检查/回滚/发布门控 |
| 3 | **Browser Research** | 深度网络研究/比较/采购/证据收集 | 隔离子 agent/来源捕获/总结/引用验证 |
| 4 | **Document & Contract** | 合同审查/合规检查/文档分析/条款提取/红线/执行总结 | 固定阶段/schemas/剧本/模板驱动输出 |
| 5 | **Finance & Reporting** | 财务总结/差异分析/KPI 报告/预算检查/董事会风格输出 | 结构化指标/来源对账/模板报告 |
| 6 | **Customer & Operations** | 入职/支持分类/客户健康审查/管道卫生/重复运营 | SOPs/策略检查/截止日期/升级规则 |
| 7 | **Incident & Recovery** | 中断/回归/安全事件/脆弱工作流 | 严重性/时间线/诊断/回滚/缓解/事后总结 |
| 8 | **Science & Experiment** | 文献综述/实验规划/数据集验证/分析管道/报告 | 可复制 artifact/谱系/不确定性声明/实验状态 |
| 9 | **Complex Projects** | 长期项目如公司运营/客户交付/开源维护/内部运营/研究项目 | 工作流/重复运营/KPI 追踪/决策队列/异常检测/生命周期管道/预算/升级 |

**每个专用 harness 应定义**: 触发条件/固定 vs 动态阶段/必需输入/澄清问题（如需要）/工作区或虚拟文件系统布局/结构化中间 schemas/每阶段验证检查/最终输出和模板/批准门/重试和回退逻辑/停止条件/记忆更新/该 harness 的 evals

## 推荐默认实现选择（30 条）

### 控制平面架构
- **混合控制平面**: REST 用于 CRUD/仪表板/历史/管理操作/外部集成，WebSockets/流通道用于实时输出/任务分发/干预/警报/机器存在

### 执行拓扑
- **Hub-and-Worker**: 持久队列和策略在 hub，工具执行在靠近真实机器环境的 worker

### 队列持久化
- 调度前持久化任务到真实存储
- 偏好显式 `goal → task graph → task assignment → result` 生命周期
- 不依赖内存消息 alone 作为队列

### 数据库选择
- 单服务器控制平面从 SQLite WAL 模式开始
- 仅并发/托管/规模真正需要时移动到 Postgres

### 状态分割
- 操作索引和协调状态在结构化存储（任务/会话/agent/批准/预算/指标/事件/信任分数）
- 规范每项目状态在 Markdown/可见文件（计划/任务/知识/决策/合同/状态/交接/失败笔记/工件/运行手册）

### 轮询模型
- 持久 worker 默认每 30 秒拉取任务认领
- 仅推送通知作为优化非唯一机制

### 任务锁定
- 调度前原子锁定任务
- 仅锁定待处理任务
- 仅完成/显式失败/超时处理时解锁

### Worktree 优先并行编码隔离
- Git 可用时偏好每并行编码任务/子任务/机器拥有车道一个 worktree
- 仅序列化工作用共享工作树

### 任务 Schema
- 给任务不止描述，包括字段：scope/mindset/context/skill tags/priority/risk level/budget/attempts/verification plan/artifacts

### 会话可见性
- 每 agent 运行应创建人类事后可检查的可见会话

### 任务执行超时
- 默认硬任务超时约 30 分钟除非任务显式证明更长合理

### 委托深度
- 默认最大子委托深度约 5

### 重试策略
- 普通执行失败自动重试一次
- 然后改变策略或升级

### 心跳和唤醒逻辑
- 长期目标用心跳
- 追踪编排器活跃度和重连时重分发停滞工作

### 离线缓冲
- Worker 与 hub 断开时本地持久出站消息到磁盘并重连时刷新
- 队列大小硬上限

### 负载均衡
- 从非常简单最少忙分数开始：活跃 agent 计数权重高/CPU 权重次要

### 批准模型
- 调度前应用批准门非仅执行后
- 结合：显式用户创建批准规则 + 基于任务内容和风险的自动决策层

### 信任模型
- 每用户和每技能或领域追踪信任非仅全局
- 基于真实任务结果提升自主

### 预算模型
- 多层追踪预算：每任务/每目标/每机器或 worker/每月
- 超时自动暂停或要求批准

### 浏览器和桌面 QA
- 浏览器流用专用怀疑 QA 评估器
- 分离构建者和评估者

### Profile 路由
- 按技能标签路由任务到 profile 特定 prompts 和模型选择

### 最近上下文和工作区默认
- 记住最近项目文件夹和机器 home/默认文件夹

### 人类可读进度镜像
- 镜像目标状态到人类可读 Markdown 或仪表板工件

### 自改进循环默认
- 一次运行一个有界改进→提交候选变更→评估→改进保留→回归回滚→定期 full eval 期间 delta eval

### 等分打破
- 变更保持相同分数时偏好更简单系统

### 主动监控
- 持续扫描实时项目：阻塞任务/太多进行中任务/陈旧交接/待决策/失败健康端点/KPI 漂移/脏仓库

### 业务和科学控制文件
- 鼓励持久文件：计划/决策/KPIs/交接/合同/运行手册/实验记录

### 上下文快照
- 每目标构建紧凑上下文快照：目标描述/任务状态总结/活跃 agent/最近改进/共享决策/预算状态

### 优雅降级
- 可选依赖（PTY/浏览器工具/外部 API）不可用时干净降级非崩溃整个 agent 平台

### 安全默认
- 存储提供商密钥或秘密时加密存储

### 机器本地执行
- 机器特定执行在有机器需要文件/认证/浏览器 profile/桌面会话的机器上

## 文件系统优先项目操作系统

**规则简单**: 任何兼容 agent 应能进入文件夹→检查文件→理解当前状态→继续工作→留文件夹在更好更新状态

### 规范文件包

| 文件 | 用途 |
|------|------|
| `project.md` / `charter.md` | 项目章程 |
| `plan.md` | 计划 |
| `tasks.md` + `tasks/` | 任务 |
| `knowledge.md` | 知识 |
| `decisions.md` | 决策 |
| `status.md` | 状态 |
| `handoff.md` | 交接 |
| `FAILURE.md` | 失败记录 |
| `artifacts/` | 工件 |
| `evals/` | 评估 |
| `runs/` / `logs/` | 运行日志 |

### Agent 规则

- 行动前阅读
- 执行中更新非仅结束
- 产生时写证据和工件
- 方向变化时记录决策
- 重要尝试失败时记录失败
- 留显式交接带下一行动/阻塞者/开放问题
