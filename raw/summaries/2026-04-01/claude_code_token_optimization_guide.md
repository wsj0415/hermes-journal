## 🎯 核心问题

最近，很多人分享了 Claude 计划的剧烈变化，指出要么是每个美元获得的 token 减少了，要么是影响计划的主要 bug。

**本指南目的：** 帮助你构建和创造更多，不浪费 token。

## 💡 关键发现

### 图片破坏缓存

在研究过程中发现：**在提示词中添加图片会破坏缓存。**

由于很多人用 Claude 做 UI/UX，不断提供图片，他们正在破坏提示词缓存。

**缓存定价数学：**
- 缓存提示词成本：仅为常规提示词的 **10%**
- 写入缓存成本：基础输入价格的 **25%**

**示例计算（Sonnet 4.6）：**
```
100 万 token 成本：$5
缓存写入：$5 × 1.25 = $6.25（仅第一次）
缓存读取：$5 × 0.1 = $0.50（每次后续）

如果缓存被破坏：
每次都要支付 $6.25，而非 $0.50
```

### 1M 上下文窗口的问题

1M 上下文窗口带来的问题比帮助更多，这是很多人没有意识到的。

**支持模型：** Claude Opus 4.6 和 Sonnet 4.6

**级别：** low、medium、high、max

## Level 3: 缓存精通 ⭐⭐⭐

**10 分钟正确设置。这才是真正省钱的地方。**

### 缓存定价基础

| 操作 | 成本 | 说明 |
|------|------|------|
| **缓存写入** | 1.25x 基础输入价格 | 仅第一次请求 |
| **缓存读取** | 0.1x 基础输入价格 | 每次后续命中 |
| **1 小时缓存** | 2x 写入成本 | 读取仍为 0.1x |

**示例（Sonnet 4.6，$3/MTok 输入）：**
- 缓存读取成本：$0.30/MTok
- 如果你每次请求发送相同的 10K-token 系统提示词，缓存在第二次调用后回本

---

### Tip 17: 启用自动缓存

**最简单起点：**
```json
{
  "model": "claude-sonnet-4-6",
  "max_tokens": 1024,
  "cache_control": {"type": "ephemeral"},
  "system": "Your system prompt here...",
  "messages": [...]
}
```

---

### Tip 18: 为可缓存性构建提示词

**Claude 处理顺序：** tools → system prompt → messages

**建议：** 把最稳定、可复用的内容放在最前面

**完美缓存候选：** 系统提示词和工具定义（很少变化）

---

### Tip 19: 理解定价数学

**计算示例：**
```
Sonnet 4.6 输入价格：$3/MTok
缓存写入：$3 × 1.25 = $3.75/MTok（第一次）
缓存读取：$3 × 0.1 = $0.30/MTok（后续）

发送 10 次相同 10K-token 系统提示词：
无缓存：10 × 10K × $3/MTok = $0.30
有缓存：$0.0375 + 9 × $0.003 = $0.0645
节省：78%
```

---

### Tip 20: 使用显式缓存断点进行细粒度控制

**适用：** 具有不同变化频率部分的复杂提示词

**限制：** 最多可使用 4 个断点

```json
{
  "role": "system",
  "content": [
    {
      "type": "text",
      "text": "Your large static instructions...",
      "cache_control": {"type": "ephemeral"}
    }
  ]
}
```

---

### Tip 21: 满足最小 token 阈值

**各模型最低要求：**

| 模型 | 最低 token |
|------|-----------|
| Opus 4.6/4.5 | 4,096 |
| Sonnet 4.6 | 2,048 |
| Sonnet 4.5/4 | 1,024 |
| Haiku 4.5 | 4,096 |

**重要：** 低于阈值，请求成功但无缓存，无错误返回

**验证：** 检查响应中的 `cache_creation_input_tokens` 字段

---

### Tip 22: 保持缓存温暖

**默认 TTL：** 5 分钟，每次命中刷新

**问题：** 如果请求间隔超过 5 分钟，缓存过期，再次支付写入成本

**解决方案：** 对于间隔较长的工作负载，使用 1 小时选项（2x 写入，读取仍 0.1x）

**建议：** 对于经常超过 5 分钟的扩展思考任务，1 小时缓存几乎总是值得

---

### ⚠️ 如何破坏你的缓存（不要做这些）

#### Tip 23: 不要在每次请求都变化的内容上放置 cache_control

**最常见错误：** 如果将缓存断点放在包含时间戳、会话 ID 或用户消息的块上，前缀哈希每次都会变化。

**结果：** 每次请求都获得缓存写入，零读取

**代价：** 重复支付 1.25x 写入溢价，永不见节省

---

#### Tip 24: 不要添加或移除图片

**重要：** 在提示词的任何地方添加或移除图片（不仅在缓存部分）都会破坏缓存。

**建议：** 如果工作流有时包含图片有时不包含，将它们视为具有单独缓存生命周期的独立请求模式。

---

#### Tip 25: 不要修改缓存断点前的任何内容

**原则：** 缓存匹配精确前缀

**影响：** 即使系统提示词、工具定义或缓存块前任何消息中的单个字符变化，也会强制完全缓存未命中和重写。

**建议：** 仔细版本化你的系统提示词

---

#### Tip 26: 不要切换思考参数

**影响：** 在请求之间切换扩展思考开/关或更改预算分配会使消息缓存断点无效。

**缓存存活：** 缓存的系统提示词和工具定义在思考变化中存活

**缓存破坏：** 消息级缓存不存活

---

#### 缓存快速参考

**破坏缓存：**
- ❌ 更改断点前内容
- ❌ 修改 tool_choice
- ❌ 添加/移除图片
- ❌ 更改思考参数
- ❌ TTL 过期

**缓存存活：**
- ✅ 更改断点后消息
- ✅ 更改 max_tokens
- ✅ 在末尾添加新用户消息

---

#### Tip 27: 注意并发请求

**问题：** 缓存条目仅在第一个响应开始流式传输后才可用

**场景：** 如果你同时发送 10 个并行请求，只有第一个写入缓存。其他 9 个不会获得缓存命中，因为条目不存在。

**建议：** 对于扇出模式，发送第一个请求，等待它开始流式传输，然后发送其余的。

## Level 4: 系统设计 ⭐⭐⭐⭐

**15 分钟架构。这些是将爱好者与生产团队分开的决策。**

### Tip 28: 构建模型路由器

**原则：** 不要将每个请求发送到同一模型

**路由策略：**
| 任务复杂度 | 模型 | 价格 |
|------------|------|------|
| 简单 | Haiku | $1/$5 |
| 中等 | Sonnet | $3/$15 |
| 复杂 | Opus | $5/$25 |

**影响：** 这单一架构决策通常是规模上最大的成本杠杆

---

### Tip 29: 对非紧急工作使用 Batch API

**折扣：** 所有 token 价格 **50% 折扣**

**时效：** 24 小时内结果

**适用：** 批量代码分析、内容生成、数据处理、任何不需要实时响应的工作负载

**组合：** 与提示词缓存叠加可实现高达 **95% 总节省**

---

### Tip 30: 使用 token 高效的工具使用

**默认支持：** 所有 Claude 4 模型默认支持 token 高效工具使用

**节省：** 平均 14% 输出 token 节省（最高 70%）

**Claude 3.7 Sonnet：** 添加 beta 头 `token-efficient-tools-2025-02-19`

**说明：** 这实际上是免费节省

---

### Tip 31: 只包含你需要的工具

**问题：** 每个工具定义都随每个请求发送

**示例：** 如果你有 15 个工具定义但给定任务只需要 2 个，你每次 API 调用都在为 13 个无用工具定义付费

**解决方案：** 基于任务动态加载工具

---

### Tip 32: 不相关任务之间清除上下文

**Claude Code：** 切换主题时使用 `/clear`

**API 集成：** 开始新对话

**问题：** 之前任务的陈旧上下文在每次后续消息中被重新处理，指数级增加成本

---

### Tip 33: 摘要长对话

**方法：** 定期将早期轮次压缩为摘要

**示例：** 用 200 字摘要替换 20 轮来回

**效果：** token 使用从指数级变为线性

---

### Tip 34: 修剪不相关轮次

**原则：** 多轮对话中不是每条消息都需要保留在上下文中

**建议：** 删除不再与当前任务相关的轮次

**保留：** 仅保留 Claude 维持连贯性所需的内容

---

### Tip 35: 将详细操作委托给子代理

**详细操作：** 运行测试、获取文档、处理日志文件

**方法：** 委托给子代理，详细输出保留在子代理上下文中

**效果：** 只有简洁摘要返回主对话

**数据：** 在 Claude Code 中，代理团队使用约 7 倍于标准会话的 token，但 token 隔离意味着主上下文保持精简和便宜

---

### 1M 上下文窗口的悖论

**问题：** 1M 上下文窗口在没有大多数用户意识到的情况下确实造成成本效率问题

**原因：**
- 每条后续消息重新处理整个上下文
- 10 轮 100K 上下文对话 = 1M token 输入
- 指数级增长，而非线性

**解决方案：** Tip 32-35（清除、摘要、修剪、委托）

**好处：** 可以记住更多
