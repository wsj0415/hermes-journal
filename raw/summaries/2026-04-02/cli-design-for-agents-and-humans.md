## 🎯 核心洞察

> **"今年构建的每个 CLI 在某个时候都会被代理调用。大多数还没准备好。"**

**问题：**
- 交互式提示
- 彩色输出
- 终端 UI

**这些都是 CLI 设计的标准假设，但在自动化代理尝试解析结果的瞬间就崩溃了。**

**但是剥离这些功能会让 CLI 对人类体验变差。**

**解决方案：** 你不需要在一个受众和另一个之间选择。你可以设计同时服务于两者的 CLI。

---

## 📋 核心哲学

### 数据与表现分离

```
┌─────────────────────────────────────────┐
│              CLI 引擎                    │
│    (内部逻辑，发射数据)                   │
└─────────────────┬───────────────────────┘
                  │
         ┌────────┴────────┐
         ↓                 ↓
   ┌──────────┐      ┌──────────┐
   │  人类     │      │   代理    │
   │  TUI     │      │  JSON    │
   │  可读格式  │      │  结构化   │
   └──────────┘      └──────────┘
```

**同一个 `watch` 命令应该：**
- 给人类：实时更新的 TUI
- 给代理：NDJSON 事件流

**你可以在不维护两个独立代码库的情况下实现这种无缝的双受众体验。**

## 🛠️ 7 个设计模式

### 1️⃣ 结构化可发现性

**入口点应该清晰映射功能。**

#### --help 优化

**分组命令** — 按功能分组，而非字母顺序：
```
❌ 字母顺序：
  add, config, delete, get, list, remove, update

✅ 功能分组：
  Task Management: add, delete, list
  Information: get, show
  Configuration: config, init
```

**明确标记入口点：**
```
Commands:
  init (start here)          Initialize a new project
  list (typical first step)  List all tasks
  get                        Get task details
```

#### 每个命令填充三个字段

| 字段 | 说明 | 示例 |
|------|------|------|
| **Short** | 一行摘要，5-10 词，以动词开头 | "List all tasks in current project" |
| **Long** | 详细说明：做什么、为什么用、与类似命令的区别 | "Lists all tasks with their current status. Use this to get an overview before drilling into specific tasks. Differs from 'show' which shows details of a single task." |
| **Example** | 3-5 个具体可复制粘贴的示例 | `cli list --status=pending`\n`cli list --assignee=me`\n`cli list --limit=10` |

**示例比描述更重要。** 开发者会在阅读前复制粘贴。代理会解析示例来推断标志模式。

---

#### 外部化代理指令

**可发现性超越 `--help`。**

**不要假设 LLM 原生知道如何使用你的工具 — 明确教它。**

**在仓库根目录发布 `AGENTS.md` 文件：**
- 定义参与规则
- 默认工作流
- 架构标准

**对于复杂命令，包含 `skills/` 目录：**
- 包含专用提示文件
- 外部代理可直接摄取

---

### 2️⃣ 代理优先互操作性

**CLI 必须是可解析和可预测的。**

#### --json 支持所有命令

**每个产生数据的命令都应该支持 `--json` 或 `--no-tui` 标志。**

```bash
# 人类使用
$ cli watch

# 代理使用
$ cli watch --json
# 或
$ cli watch --no-tui
```

**输出应该是有效的 JSON 或 NDJSON。**

**如果代理无法解析你的输出，你的 CLI 在代理世界中不存在。**

---

#### 自动检测受众

**支持环境变量：**
```bash
NO_COLOR
[APP]_NO_TUI
```

**当这些设置时，或 stdout 被管道时，完全跳过交互元素：**
- ❌ 无提示
- ❌ 无旋转器
- ❌ 无颜色代码

---

#### 非交互回退

**使用 TUI 的命令（如 Bubble Tea）应该有纯模式：**
```
TUI = 人类界面
JSON 输出 = 代理界面
同一命令，两个渲染器
```

---

#### 保护上下文窗口

**代理有严格的 token 限制。**

**主动截断大文本块：**
```bash
# 默认：截断输出
$ cli logs
[... truncated, 50 lines shown ...]

# 明确选择：完整输出
$ cli logs --full
```

**屏蔽敏感密钥：**
```bash
# 默认：掩码
API_KEY: sk-****

# 明确选择：显示
$ cli config --show-secrets
```

---

#### 预处理和排序数据

**代理不应该编写复杂的数据操作逻辑来查找重要内容。**

**自动预排序 CLI 输出：**
```
✅ 严重漏洞
✅ 无限制密钥
✅ 待处理任务
```

**最关键、可操作的项目总是出现在响应顶部。**

---

#### 委托状态管理

**不要在 CLI 进程中将代理困在交互状态循环中。**

**保持 CLI 完全无状态：**
```bash
# 使用引用标识符
$ cli get --task <ID>
$ cli update --task <ID> --status=done
```

**让后端服务维护长期 token 上下文和对话历史，CLI 仅作为快速传输机制。**

---

### 3️⃣ 配置和上下文

**CLI 应该理解其环境，无需在每次调用时需要过多标志。**

#### 遵循 XDG

```
✅ ~/.config/app/config.yaml
❌ ~/.app  # 主目录中无点文件
❌ ~/.app/  # 无专有格式的隐藏文件夹
```

#### 命名环境

**支持多个配置（local、staging、prod），带简单的 `--env` 标志：**
```bash
$ cli deploy --env prod
$ cli deploy --env staging
```

**这对代理特别重要。** 编排器可以设置 `--env prod`，无需知道 URL 或 token。

---

### 4️⃣ 错误指导

**不要只报告错误。提供解决路径。**

#### 上下文提示

**当命令因缺少先决条件而失败时，包含 `Hint:` 行：**
```
Error: Missing API key

Hint: Run `cli config init` to set up your API key.
      Or set the API_KEY environment variable.
```

**代理解析这些提示来自我纠正。人类感激不必搜索文档。**

---

#### 快速失败

**在执行重逻辑之前验证配置和连接性。**

```
❌ 让代理等待 30 秒 API 调用后才发现 auth token 缺失
✅ 首先验证 auth，然后执行
```

---

#### 确定性退出码

| 退出码 | 含义 |
|--------|------|
| **0** | 成功 |
| **1** | 一般错误 |
| **2** | 无效用法/错误标志 |
| **3** | 连接/auth 失败 |

**如果你的 CLI 在失败操作上返回 0，因为"命令本身运行了"，你会破坏调用它的每个自动化。**

---

### 5️⃣ 标志和参数一致性

**CLI 语法应该天生可预测。**

#### 标准化缩写

```
✅ 一致：
  -o = --out-dir (所有命令)
  -f = --force (所有命令)

❌ 不一致：
  命令 1: -o = --out-dir
  命令 2: -o = --output  # 打破代理推理
```

**如果一个用户（或代理）学会如何使用一个命令，该直觉应该无缝延续到其余应用。**

---

#### 位置 vs 可选

```
✅ 位置参数：核心、必需实体
  $ cli get <task-id>
  $ cli delete <user-id>

✅ --flags：严格用于可选修改器
  $ cli list --limit=10
  $ cli list --status=pending
```

---

#### 安全默认值

**默认行为应该总是最安全、最常见的路径。**

```
✅ 安全默认：
  $ cli delete <id>  # 需要确认

✅ 显式标志用于破坏性操作：
  $ cli delete <id> --force  # 跳过确认
```

---

### 6️⃣ 终端视觉设计

**颜色应该服务于功能目的，而非美学目的。**

#### 语义颜色令牌

**使用基于意义的令牌，而非原始颜色名称：**

| 令牌 | 用途 | 示例 |
|------|------|------|
| **Accent** | 地标：标题、组标题、部分标签 | 部分标题 |
| **Command** | 扫描目标：命令名、标志 | 命令名 |
| **Pass** | 成功：已完成任务、成功状态 | ✅ |
| **Warn** | 瞬态：活动任务、警告、待处理状态 | ⚠️ |
| **Fail** | 错误：失败任务、错误、拒绝状态 | ❌ |
| **Muted** | 去强调：元数据、类型、默认值、预览 | 灰色文本 |
| **ID** | 标识符：唯一 ID（任务 ID、技能 ID） | 任务 ID |

---

#### 经验法则

| 规则 | 说明 |
|------|------|
| **保留颜色用于状态** | 绿色=成功，黄色=警告，红色=错误，灰色=元数据 |
| **不要着色描述或帮助文本** | 过度着色导致彩虹输出，什么都不突出 |
| **空白优于颜色用于层次** | 定位和对齐比颜色更好地传达结构 |
| **支持浅色和深色终端** | 使用自适应颜色，无论用户主题如何保持对比 |

---

### 7️⃣ 版本控制和生命周期

**CLI 工具不是静态工件。**

它在动态序列中运行：
- 版本漂移
- 操作突然中断
- 安全工作流需要审计追踪

**智能处理这些生命周期事件可防止自动化系统中的静默失败。**

#### 版本感知

```bash
$ cli --version
cli version 1.2.3

# 可选：通知用户自上次运行以来的重大更新
```

#### 优雅处理 SIGINT

```bash
# Ctrl+C 时
✅ 无数据损坏
✅ 清理临时文件
✅ 保存进度状态
```

#### 跟踪"Actor"

**在写入操作上跟踪执行者：**
```bash
# 从 git config user.name 或 $USER 派生
Actor: john.doe@example.com
Timestamp: 2026-03-31T00:39:10Z
Operation: task.create
```

**用于审计。**

## 🎬 端到端演示：YouTube CLI

**YouTube CLI 是一个典型示例。**

你现在可以与代理协作管理和审查你的 YouTube 频道。

**通过利用代理感知 CLI 技能，你和代理可以构建所需的 exact 工具集：**
- 检索频道统计
- 切换视频可见性
- 上传新内容

**以前，这需要：**
- 寻找特定的 MCP 服务器
- 或手动解析密集的 YouTube API 文档

**现在，只需：**
```bash
$ yt-cli stats
$ yt-cli upload video.mp4 --title="New Video"
$ yt-cli set-visibility <video-id> --public
```

---

### 关键要点

**对于 AI 代理可靠地执行这样的工作流，底层 CLI 必须为其设计。**

**如果 `yt-cli` 用交互提示阻止代理：**
```
❌ "Are you sure you want to upload? (Y/n)"
❌ 返回不可解析的彩色终端文本
```

**整个自动化将失败。**

## 💡 关键洞察

### 洞察 1：双受众设计

| 特性 | 人类需求 | 代理需求 |
|------|----------|----------|
| **输出格式** | TUI、可读 | JSON、结构化 |
| **交互** | 提示、确认 | 无交互、可管道 |
| **颜色** | 美观、语义 | 无颜色、或 NO_COLOR |
| **错误** | 友好消息 | 退出码 + 提示 |

**解决方案：** 同一命令，两个渲染器。

---

### 洞察 2：代理优先设计

**2026 年最好的 CLI 不是那些有最漂亮终端界面的。**

**而是那些在人类输入命令和代理编程调用时同样工作的。**

**设计原则：**
1. `--json` 支持所有命令
2. 自动检测受众
3. 保护上下文窗口
4. 预处理和排序数据
5. 委托状态管理

---

### 洞察 3：错误即指导

**不要只报告错误。提供解决路径。**

```
❌ 传统：
Error: Missing API key

✅ 代理友好：
Error: Missing API key

Hint: Run `cli config init` to set up your API key.
      Or set the API_KEY environment variable.

Exit code: 3
```

**代理解析这些提示来自我纠正。**

---

### 洞察 4：颜色语义化

**颜色应该服务于功能目的，而非美学目的。**

| 原则 | 说明 |
|------|------|
| 保留颜色用于状态 | 绿=成功，黄=警告，红=错误 |
| 不要着色描述 | 过度着色导致什么都不突出 |
| 空白优于颜色 | 定位和对齐传达结构 |
| 支持浅色和深色 | 自适应颜色保持对比 |
