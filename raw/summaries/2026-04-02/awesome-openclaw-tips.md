## 💬 消息 (Messages)

### MES-01: 启用确认反应

**问题：** 默认情况下，确认反应范围有限。

**解决：** 显式设置 `messages.ackReactionScope`。

```json
{
  "messages": {
    "ackReactionScope": "all"
  }
}
```

**有效值：**
- `group-mentions` — 仅群提及
- `group-all` — 仅群所有
- `direct` — 仅私聊
- `all` — 所有聊天 ✅ 推荐
- `off` / `none` — 关闭

**效果：** 确认反应在更广泛的聊天中出现，而非仅限私聊。

## 📨 Telegram

### TEL-01: 使用 Telegram 内联按钮

**问题：** 用户必须反复输入相同的回复、批准或短命令，工作流比需要的慢。

**解决：** OpenClaw 支持 Telegram 内联按钮，重复操作可以一键完成而非再次输入。

**配置：**
```json
{
  "channels": {
    "telegram": {
      "capabilities": {
        "inlineButtons": "all"
      }
    }
  }
}
```

**支持范围：** `off`, `dm`, `group`, `all`, `allowlist`

**适用场景：**
- ✅ 批准和审查工作流
- ✅ 快速回复
- ✅ 常见后续操作
- ✅ 任何重复选择作为点击比输入更清晰的地方

**发送测试命令：**
```bash
openclaw message send \
  --channel telegram \
  --account <accountId> \
  --target <chatId> \
  --message "<text>" \
  --buttons '<json>'
```

**按钮 JSON 格式：**
```json
[
  [
    {"text": "批准", "callback_data": "approve"},
    {"text": "拒绝", "callback_data": "reject"}
  ]
]
```

## 🧠 记忆 (Memory)

### MEM-01: 让代理从错误中学习

**问题：** 默认情况下，每次会话都是全新的开始。代理不记得上次尝试失败或你纠正它的事情。两周后，它仍在重复第三天的错误。

**解决：** 在工作区创建 `.learnings/` 文件夹。

**文件结构：**
```
.learnings/
├── ERRORS.md      # 命令失败、故障、异常
└── LEARNINGS.md   # 纠正、知识差距、工作流发现
```

**更新 SOUL.md：**
```markdown
在开始任何任务前，检查 .learnings/ 查找相关过去错误。
当你失败某事或我纠正你时，立即记录到 .learnings/。
```

**随时间推移：** 重复的课程可以提升到 SOUL.md 本身，以便它们在未来会话中自动应用。

**完整版本：** 安装 [self-improving-agent skill](https://clawhub.ai/pskoett/self-improving-agent) 获取自动钩子、结构化条目和提升工作流。

### MEM-02: 在压缩吞噬前刷新重要状态

**问题：** 长 OpenClaw 会话不会无限增长。一旦上下文满了，OpenClaw 自动压缩会话。这通常意味着摘要幸存，加上最近的轮次。任何仅存在于聊天中的重要内容现在处于风险中。

**解决：** 将压缩视为最后期限。重要状态应该在发生前已经写入磁盘。

**启用内置预压缩内存刷新：**
```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "memoryFlush": {
          "enabled": true,
          "softThresholdTokens": 4000
        }
      }
    }
  }
}
```

**工作原理：**
- 监控上下文使用
- 在硬压缩前触发静默刷新
- 将持久状态写入工作区
- 每个压缩周期运行一次
- 使用 NO_REPLY，不骚扰用户

**自然配对：** MEM-01。聊天是工作记忆。文件是记忆。

### MEM-03: 在付费嵌入前使用 SQLite 记忆搜索

**问题：** 大多数个人 OpenClaw 设置不需要向量数据库来查找旧笔记。

**解决：** 使用带有 FTS5 的本地 SQLite 索引。

**构建数据库：**
```bash
node tips/mem-03/rebuild-db.js
```

**搜索：**
```bash
node tips/mem-03/relevant-memory.js "关于之前工作的查询"
```

**工作原理：**
- 脚本扫描 markdown 文件
- 构建 SQLite 数据库
- 使用全文搜索快速发现可能匹配
- 无 API 成本
- 无嵌入管道
- 无外部服务

### MEM-04: 将聊天历史视为缓存，而非真相源

**问题：** 如果你希望 OpenClaw 可靠地记住事情，将它们写入 OpenClaw 已经知道如何使用的文件。转录记忆是脆弱的。压缩、重置和长空闲间隔都使聊天成为保存重要内容的糟糕地方。

**解决：** OpenClaw 已经有用于此的真实文件模式：

| 文件 | 用途 |
|------|------|
| **MEMORY.md** | 持久事实、偏好、决策 |
| **memory/YYYY-MM-DD.md** | 日常笔记和运行上下文 |
| **AGENTS.md** | 代理应如何使用这些文件的常设规则 |

**规则：**
- 重要事实属于 MEMORY.md
- 短期工作上下文属于日常记忆文件
- AGENTS.md 应告诉代理读取和更新它们，而非信任转录携带一切

**如果任务状态重要：** 放入实际任务系统或你已使用的其他持久存储。

**原则相同：** 如果丢失转录会破坏工作流，状态在错误的地方。

### MEM-05: 将会话拆分为线程，防止上下文跨主题泄漏

**问题：** 一个长 OpenClaw 对话变成杂物抽屉。编码、研究、行政和随机问题都混在一起，每个新轮次都拖着那个包袱前进。

**解决：** OpenClaw 已经有可使用的会话边界。

**隔离机制：**
- **群聊** — 按群隔离状态
- **Telegram 论坛主题** — 获得自己的 `:topic:<threadId>` 会话键
- **Slack 和 Discord 线程** — 被视为线程会话
- **Discord 频道** — 获得自己的隔离会话

**实际修复：** 按主题或频道拆分工作。
- 一个线程用于编码
- 一个用于研究
- 一个用于行政
- 一个用于个人操作

**聚焦线程给代理聚焦上下文**，而非一个巨大的混合转录。

**Telegram 设置：**
1. 打开 BotFather
2. 从 My bots 选择你的机器人
3. 进入 Bot Settings
4. 打开 Threaded Mode

**效果：** Telegram 给机器人单独的标签/主题 — 编码在一个主题，研究在另一个，行政在另一个 — 而非一切污染一切的混合转录。

### MEM-06: 使工作区文件夹成为真相源并放入 git

**问题：** AGENTS.md、SOUL.md、TOOLS.md、IDENTITY.md、USER.md、HEARTBEAT.md、MEMORY.md 和 memory/ 不是松散笔记。它们是操作状态。

**解决：** OpenClaw 自己的文档推荐将工作区视为私有内存并将其放入私有 git 仓库。

**好处：**
- ✅ 备份
- ✅ diff
- ✅ 清晰查看提示、记忆文件或操作规则何时更改

**实际设置：**
```bash
git init
git add AGENTS.md SOUL.md TOOLS.md IDENTITY.md USER.md HEARTBEAT.md MEMORY.md memory/
git commit -m "Add agent workspace"
```

**如果使用 GitHub 或 GitLab：** 使仓库私有并推送到那里。

**如果安装了 git：** 全新 OpenClaw 工作区甚至可以自动初始化，告诉你这不是奇怪的自定义模式 — 它是将工作区视为真相源的正常方式。

**自然配对：** MEM-04。如果工作区是真相源，git 是你保持该真相源可恢复的方式。

---

### MEM-07: 持续备份工作区，而非仅一次

**问题：** `git init` 不是备份。备份只有在持续推送更新后才变成真实的。

**解决：** OpenClaw 的工作区文档已经显示持续循环：

```bash
git status
git add .
git commit -m "Update memory"
git push
```

**这是要构建的习惯。** 提示变化。MEMORY.md 变化。memory/ 变化。如果私有仓库只获得第一次提交而从未获得后续更新，它不是真正的真相源备份。

**实际版本：**
1. 在 AGENTS.md 中放入备份策略
2. 从专用 cron 运行它

**AGENTS.md 应定义：** 什么算有意义的变化

**cron 应做：** 实际检查和同步

**这不属于 HEARTBEAT.md** — 心跳用于健康检查，而非工作区维护。

**提交和推送时机：** 有意义的 workspace 变化，而非每个小编辑。

**好触发器：**
- 提示更新
- 新持久记忆
- 工作流变化
- 任何你痛恨丢失的 workspace 内容

**示例 cron：**
```bash
openclaw cron create \
  --name workspace-backup-check \
  --cron "0 21 * * *" \
  --message "检查工作区 git 仓库。如果有意义的 workspace 文件变化，提交并推送它们。跳过琐碎噪音。永不提交秘密或 ~/.openclaw/ 下的任何内容。报告发生了什么。"
```

### MEM-08: 定期自清洁记忆，而非让它永远腐烂

**问题：** 记忆质量取决于被移除的内容和被添加的内容一样多。

**随时间推移：** MEMORY.md 收集重复、过时事实、旧偏好和矛盾。

**解决：** OpenClaw 自己的模板指导已经建议每几天进行一次维护传递：

1. 读取最近的 memory/YYYY-MM-DD.md 文件
2. 保留重要课程
3. 更新 MEMORY.md
4. 移除不再真实的过时信息

**正确心智模型：**
- **日常记忆文件** = 原始笔记
- **MEMORY.md** = 精选智慧

**如果一切永远保留：** 检索变慢，矛盾增加，质量下降。

**建议频率：** 每 3-7 天

**维护检查清单：**
- [ ] 删除重复条目
- [ ] 更新过时事实
- [ ] 移除矛盾偏好
- [ ] 合并相关条目
- [ ] 提升新课程到 SOUL.md

## 🛡️ 可靠性 (Reliability)

### REL-01: 不要将所有回退放在同一提供商

**问题：** 单点故障。

**解决：** 分散回退到不同提供商。

**示例：**
- 主模型：Claude Opus
- 回退 1: GPT-4
- 回退 2: 本地模型

---

### REL-02: 你的代理说"完成"时并非如此

**问题：** 代理自信地报告完成，即使工作未完成。

**解决：** 实施验证检查。

**验证模式：**
```
任务完成 → 验证检查 → (失败) → 重试/报告
                      ↓ (成功)
                   真正完成
```

---

### REL-03: 使用心跳旋转重复检查，而非仅重复一个通用检查

**问题：** 相同检查每次运行，效率低。

**解决：** 心跳轮换不同检查。

**示例轮换：**
- 心跳 1: 邮箱检查
- 心跳 2: 日历检查
- 心跳 3: 通知检查
- 心跳 4: 健康检查

---

### REL-04: 在坏运行燃烧数小时前打开工具循环检测

**问题：** 代理陷入工具调用循环，燃烧 token 和时间。

**解决：** 启用工具循环检测。

**配置：**
```json
{
  "agents": {
    "toolLoopDetection": {
      "enabled": true,
      "maxRepeats": 3
    }
  }
}
```

## 💸 成本 (Cost)

### COST-01: 你的心跳模型成本比你想象的更多

**问题：** 频繁心跳使用昂贵模型，成本累积。

**解决：** 对心跳使用更便宜模型。

**示例：**
- 主任务：Claude Opus
- 心跳：Claude Haiku（10x 更便宜）

---

### COST-02: 使用 cache-ttl 修剪，否则空闲会话将重新缓存垃圾历史

**问题：** 空闲会话重新缓存无用历史，浪费钱。

**解决：** 设置缓存 TTL 修剪。

**配置：**
```json
{
  "cache": {
    "ttl": "24h",
    "prune": true
  }
}
```

---

### COST-03: 本地模型通常是虚假经济

**问题：** 本地模型看似省钱，但：
- 设置成本高
- 维护时间
- 质量较低导致更多重试
- 机会成本

**解决：** 仅对真正重复机械任务使用本地模型。

---

### COST-04: 仅对重复机械任务使用本地模型

**适用场景：**
- ✅ 批量数据提取
- ✅ 简单分类任务
- ✅ 重复格式化

**不适用：**
- ❌ 复杂推理
- ❌ 创造性任务
- ❌ 关键决策

## ⚙️ 运营 (Operations)

### OPS-01: 为代理和子代理设置显式并发限制

**问题：** 无限制并发导致资源耗尽。

**解决：** 设置显式限制。

**配置：**
```json
{
  "concurrency": {
    "maxAgents": 5,
    "maxSubagents": 10
  }
}
```

## ⏱️ 自动化 (Automation)

### AUTO-01: 常设命令定义什么，cron 定义何时

**区别：**
| 特性 | 常设命令 | Cron |
|------|----------|------|
| **定义** | 做什么 | 何时做 |
| **触发** | 事件驱动 | 时间驱动 |
| **示例** | "监控邮箱" | "每 30 分钟检查" |

---

### AUTO-02: 对嘈杂杂务使用隔离 cron 作业

**问题：** 嘈杂杂务污染主会话历史。

**解决：** 使用隔离 cron 作业。

**配置：**
```json
{
  "cron": {
    "isolated": true,
    "sessionTarget": "isolated"
  }
}
```

**好处：**
- ✅ 主会话保持干净
- ✅ 独立执行上下文
- ✅ 易于调试

## 🏗️ 架构 (Architecture)

### ARCH-01: 停止使用一个通用代理做所有事情

**问题：** 通用代理在所有事情上平庸。

**解决：** 专用代理用于专用任务。

**示例团队：**
| 代理 | 职责 | 模型 |
|------|------|------|
| **研究员** | 网络搜索、数据收集 | Haiku |
| **作家** | 内容创作、编辑 | Sonnet |
| **开发者** | 编码、调试 | Opus |
| **审查员** | 质量检查、验证 | Sonnet |

---

### ARCH-02: 保持编排器作为管理者，而非执行者

**问题：** 编排器尝试自己做所有事情，变成瓶颈。

**解决：** 编排器管理，子代理执行。

**职责分离：**
```
编排器：
- 接收任务
- 分解工作
- 分配给子代理
- 合并结果
- 报告完成

子代理：
- 执行具体任务
- 报告进度
- 请求帮助（如需要）
```

---

### ARCH-03: 给不同模型不同的提示文件

**问题：** 相同提示用于所有模型，未优化各自优势。

**解决：** 每个模型定制提示。

**文件结构：**
```
prompts/
├── opus/
│   ├── SOUL.md      # 复杂推理优化
│   └── TOOLS.md
├── sonnet/
│   ├── SOUL.md      # 平衡任务优化
│   └── TOOLS.md
└── haiku/
    ├── SOUL.md      # 快速任务优化
    └── TOOLS.md
```

---

### ARCH-04: 预定义子代理工作区，否则它们丢失共享上下文

**问题：** 子代理启动无共享上下文，重复工作。

**解决：** 预定义共享工作区。

**配置：**
```json
{
  "subagents": {
    "sharedWorkspace": true,
    "workspacePath": "./shared-workspace"
  }
}
```

**好处：**
- ✅ 共享记忆
- ✅ 共享工具
- ✅ 共享状态
- ✅ 避免重复工作

## 💡 关键洞察

| **Git 仓库** | 真相源 | 最高（版本控制） |

### 洞察 2：成本优化层次

| 策略 | 节省 | 复杂度 |
|------|------|--------|
| 心跳用便宜模型 | 50-80% | 低 |

| 本地模型用于机械任务 | 30-60% | 中 |
| 避免本地模型用于复杂任务 | 避免重试成本 | 低 |

---

### 洞察 3：可靠性层次

| 级别 | 策略 | 效果 |
|------|------|------|
| **L1** | 工具循环检测 | 防止灾难 |
| **L2** | 完成验证 | 确保质量 |
| **L3** | 心跳轮换 | 提高效率 |
| **L4** | 多提供商回退 | 防止单点故障 |

### 洞察 4：架构演进

```
阶段 1: 单通用代理
```
