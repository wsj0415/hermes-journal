## ❌ 问题：没有反思的记忆

**Claude 记得一切。**
- 它写自己的记忆
- 读取你的 CLAUDE.md
- 跨会话遵循你的约定

**但它不学习。**

### 典型场景

```
会话 1：捕获定价测试显示 40% 流失率
会话 2：捕获竞争对手取消了免费层级
会话 3：捕获 onboarding 用户流失较少

三个独立的观察，三个独立的会话 — Claude 从未将它们连接起来。
```

**没有反思的记忆只是存储。**

大多数 CLAUDE.md 设置正是如此 — 一个不断增长的纠正日志，从不提升模式，从不测试自己的假设，从不评估工作是否真的符合你的标准。

---

## ✅ 解决方案：三个模块

### 模块 1：知识架构 — 学习循环

#### 失败模式

Claude 捕获观察但从不重新审视它们。上周的洞察不告知本周的工作。没有机制来测试模式是否成立、在成立时提升它、或在不成立时丢弃它。

#### 核心原则

**强制主动检索** — 在每个任务之前：
1. Claude 在提出建议前读取现有知识
2. 任务后，提取学到的内容
3. 存储在具有清晰层次结构的领域特定文件夹中

#### 知识层次

```
knowledge/
├── observations/        # 原始观察
├── hypotheses/          # 需要更多数据的假设
└── rules/               # 默认应用的确认规则
```

#### 提升循环

```
观察 → 假设 → (多次确认) → 规则
                         ↓
                    (新数据矛盾)
                         ↓
                    降级回假设
```

**系统自我纠正，而非积累过时的建议。**

#### 实际成果

**一个月后：**
- 📁 一个文件夹
- 📝 24 条 Claude 自己提升的规则
- 🎯 我可从未提示过这些
- 📈 从数十次会话中浮现

**Claude 明显更好 — 不是因为更好的提示词，而是因为系统一直在学习。**

### 模块 2：决策日志 — 可审查的推理

#### 失败模式

```
有人问：为什么选择 Postgres 而非 DynamoDB？
或者：为什么取消了免费层级？

没人记得。

所以你花 30 分钟重新辩论一个已经做出的决定。
```

#### 核心原则

**决策前：**
1. Claude 首先搜索该领域的先前决策
2. 如果存在，遵循现有推理（除非新信息使其无效）
3. 如果不存在，记录完整上下文

#### 决策日志模板

```markdown
## 决策：[决策标题]

**日期：** YYYY-MM-DD

**决定内容：** [决定了什么]

**考虑的替代方案：**
- 选项 A：[描述]
- 选项 B：[描述]

**为什么选择这个选项：**
[详细推理]

**接受的权衡：**
- [权衡 1]
- [权衡 2]

**相关上下文：**
[任何影响决策的因素]
```

#### 意外发现

**当决策被替换时：**
- 日志创建可追踪的链条
- 你可以看到推理随时间演变

**作者的洞察：**
> "我最有信心的决策命中率最差。那些我强迫自己思考替代方案的决策 80% 是正确的。"

### 模块 3：质量门 — 可复合的评估标准

#### 失败模式

```
Claude 完成任务 → 告诉你完成了
你问：工作好吗？
Claude：是的 — 自信地，每次都这样
```

**问题：** 构建东西的代理无法客观判断这个东西。

**Anthropic 工程团队已记录：** 代理自信地赞扬他们生产的工作，即使质量平庸。

#### 核心原则

**给 Claude 具体的、可测试的评估标准**，特定于你的项目。

**不是模糊的愿望：**
- ❌ "要彻底"
- ❌ "要高质量"

**而是具体的检查：**
- ✅ "为月度和年度队列建模流失影响"
- ✅ "记录每个用户流的边缘情况"
- ✅ "包含至少 3 个竞争对手参考"

#### 标准演变

```
检查捕获真实问题 → 被记录
     ↓
检查持续捕获同类问题 → 提升为自动门
     ↓
检查从未触发 → 标记修剪
```

**质量门槛围绕你的实际失败模式收紧，而非冻结在 wiki 中积灰。**

**如果你用过 Scrum 的 Definition of Done，这就是那个 — 但它每周变得更犀利。**

```
      ↓
规则塑造未来决策
      ↓

      ↓
标准反馈到知识
      ↓
(循环继续)
```

**你粘贴三个模块。系统从你的实际工作中自我构建。**

## 💡 关键洞察

### 洞察 1：记忆 ≠ 学习

| 特性 | 记忆 | 学习 |
|------|------|------|
| **定义** | 捕获观察 | 连接观察 |
| **行为** | 存储事实 | 测试假设 |
| **结果** | 纠正日志 | 浮现规则 |
| **价值** | 低 | 高 |

**大多数 CLAUDE.md 只有记忆。这个系统添加学习。**

---

### 洞察 2：决策可追溯性

**问题：** 团队重新辩论已决定的事

**解决：** 决策日志创建可追溯链条

**好处：**
- ✅ 避免重新辩论
- ✅ 看到推理演变
- ✅ 新团队成员快速上手

---

### 洞察 3：质量门进化

**传统方法：** 静态 Definition of Done

**这个方法：** 动态质量门，每周变犀利

**机制：**
- 捕获问题 → 记录
- 持续捕获 → 提升为自动门
- 从未触发 → 标记修剪

**结果：** 质量门槛围绕实际失败模式收紧

---

### 洞察 4：系统自我构建

**最强大的部分：**

> "一个月后，Claude 应用了 24 条它自己编写的项目特定规则 — 这些规则我从未提示过。"

**不是作者写的规则。是从数十次会话的模式中浮现的。**

**这才是真正的自改进系统。**
