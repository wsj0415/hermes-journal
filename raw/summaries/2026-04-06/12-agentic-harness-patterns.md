## 🎯 核心概念

**什么是 Agentic Harness？**

代理 Harness 是构建代理应用程序时的可复用设计模式集合。这些模式来自对 Claude Code 泄露代码的分析，代表了代理设计的基石级架构决策。

**4 个类别：**
1. **Memory & Context**（记忆与上下文）— 5 个模式
2. **Workflow & Orchestration**（工作流与编排）— 3 个模式
3. **Tools & Permissions**（工具与权限）— 3 个模式
4. **Automation**（自动化）— 1 个模式

## 📚 类别 1：Memory & Context（5 个模式）

### 模式 1：Project Instructions（项目指令）

**问题：**
```
没有持久指令文件 → 每次会话从零开始
→ 用户重复相同约定
→ 代理在第 5 次会话犯第 1 次的错误
```

**解决：**
- 引入持久项目级配置文件
- 每次会话开始自动加载
- 定义构建命令、测试命令、架构规则、命名约定、编码标准
- 随仓库一起分发，而非用户剪贴板

**何时使用：**
- ✅ 代理跨多个会话在代码库上工作
- ⚠️ 权衡：维护负担（文件必须随项目演进而更新）

**视觉：**
```
┌─────────────────────────┐
│   CLAUDE.md / AGENTS.md │
│   - 构建命令             │
│   - 测试命令             │
│   - 架构规则             │
│   - 命名约定             │
│   - 编码标准             │
└─────────────────────────┘
        ↓ 自动加载
┌─────────────────────────┐
│     Agent Session       │
└─────────────────────────┘
```

---

### 模式 2：Scoped Instructions（范围指令）

**问题：**
```
单指令文件 → 小项目可行
大代码库 → 要么变成被忽视的巨大 blob
         要么保持太通用对任何目录都无用
```

**解决：**
- 从多个文件动态加载指令，不同范围：
  - 组织级
  - 用户级
  - 项目根目录
  - 父目录
  - 子目录
- 代理看到不同规则取决于在代码库何处工作
- Import 语法允许跨文件拆分大指令集无重复

**何时使用：**
- ✅ Monorepo、多语言项目、不同目录遵循不同约定
- ⚠️ 权衡：可发现性（指令分布在多文件中难理解）

**视觉：**
```
┌──────────────────────────────────────┐
│  Organization Level (.org.md)        │
├──────────────────────────────────────┤
│  User Level (~/.claude/USER.md)     │
├──────────────────────────────────────┤
│  Project Root (CLAUDE.md)           │
├──────────────────────────────────────┤
│  Parent Dir (src/CLAUDE.md)         │
├──────────────────────────────────────┤
│  Child Dir (src/api/CLAUDE.md)      │
└──────────────────────────────────────┘
        ↓ 动态加载
┌─────────────────────────┐
│     Agent Context       │
└─────────────────────────┘
```

### 模式 3：Tiered Memory（分层记忆）

**问题：**
```
代理同样记住一切 → 实际上记不住任何东西
加载所有记忆到上下文窗口 → 浪费 token、命中大小限制
有用信息埋在噪音下
```

**解决：**
组织代理记忆到不同层，不同加载策略：

| 层 | 说明 | 加载策略 |
|----|------|----------|
| **索引层** | 紧凑索引（Claude Code 上限 200 行） | 始终在上下文 |
| **主题层** | 特定主题文件 | 当前任务匹配时按需加载 |
| **完整层** | 完整会话转录 | 仅在需要时搜索 |

**何时使用：**
- ✅ 代理跨多会话运行，需要保留偏好、决策、工作流状态
- ⚠️ 权衡：决定什么放哪里的复杂性、层间晋升/降级、索引同步

**视觉：**
```
┌─────────────────────────┐
│  Index (200 lines)      │ ← 始终在上下文
├─────────────────────────┤
│  Topic Files            │ ← 按需加载
│  - memory/topics/ai.md  │
│  - memory/topics/ui.md  │
├─────────────────────────┤
│  Full Transcripts       │ ← 仅搜索时
│  - memory/sessions/001  │
│  - memory/sessions/002  │
└─────────────────────────┘
```

### 模式 4：Memory Consolidation（记忆巩固）

**问题：**
```
即使有分层记忆 → 代理记忆随时间退化
重复条目积累、过时事实矛盾新事实
索引增长直到不再紧凑
```

**解决：**
- 后台进程定期审查、去重、修剪、重组代理记忆（空闲时间）
- 想象为代理状态的垃圾回收
- Claude Code 泄露代码显示"autoDream"模式：
  - 合并重复
  - 修剪矛盾
  - 保持索引紧凑

**何时使用：**
- ✅ 代理跨多会话积累记忆，无法依赖用户手动策划
- ⚠️ 权衡：巩固过程本身用 token、可能犯错、过度激进修剪可能删除用户仍需要的东西

**视觉：**
```
┌─────────────────────────┐
│   Raw Memory            │
│   - 重复条目             │
│   - 过时事实             │
│   - 矛盾信息             │
└─────────────────────────┘
        ↓ autoDream 后台进程
┌─────────────────────────┐
│   Consolidated Memory   │
│   - 去重                 │
│   - 矛盾已解决           │
│   - 索引紧凑             │
└─────────────────────────┘
```

### 模式 5：Context Compaction（上下文压缩）

**问题：**
```
长代理会话 → 最终命中上下文窗口限制
代理要么完全丢失最早上下文，要么停止工作
对需要多轮持续推理的任务都不可接受
```

**解决：**
应用多阶段压缩，针对对话不同年龄调优：

| 阶段 | 压缩级别 | 说明 |
|------|----------|------|
| **最近轮次** | 完整细节 | 无压缩 |
| **较老轮次** | 轻度摘要 | 保留关键点 |
| **很老轮次** | 激进折叠 | 仅核心事实 |

Claude Code 使用四层：
1. HISTORY_SNIP
2. Microcompact
3. CONTEXT_COLLAPSE
4. Autocompact

**何时使用：**
- ✅ 会话常规超过 20-30 轮
- ⚠️ 权衡：有损压缩（每步摘要丢弃细节，代理后需 collapsed 段内容可能幻觉而非承认忘记）

**视觉：**
```
┌─────────────────────────────────────────┐
│ Recent Turns (Full Detail)              │
│ "User asked about auth..."              │
├─────────────────────────────────────────┤
│ Older Turns (Light Summary)             │
│ "Discussed JWT middleware options..."   │
├─────────────────────────────────────────┤
│ Very Old Turns (Aggressive Collapse)    │
│ "Auth system implemented with JWT"      │
└─────────────────────────────────────────┘
```

## 🔄 类别 2：Workflow & Orchestration（3 个模式）

### 模式 6：Explore-Plan-Act（探索 - 计划 - 行动）

**核心主题：** 分离
- 读取与写分离
- 研究上下文与编辑上下文分离
- 顺序工作与并行工作分离

**问题：**
```
代理直接跳到编辑文件 → 基于不完整理解做修改
→ 编辑错误文件、错过依赖、忽略现有模式
```

**解决：**
分离工作流为三阶段，递增写权限：

| 阶段 | 权限 | 活动 |
|------|------|------|
| **Explore** | 只读 | 读取、搜索、映射代码库 |
| **Plan** | 讨论 | 与用户讨论方法 |
| **Act** | 完全工具访问 | 编辑、执行、提交 |

**何时使用：**
- ✅ 任何触及不熟悉代码库的任务、涉及跨多文件非平凡变更
- ⚠️ 权衡：速度（强制执行探索和规划增加轮次，简单任务感觉慢）

**视觉：**
```
┌─────────────┐
│   Explore   │ ← 只读：读取/搜索/映射
└─────────────┘
      ↓
┌─────────────┐
│    Plan     │ ← 讨论：与用户确认方法
└─────────────┘
      ↓
┌─────────────┐
│    Act      │ ← 完全访问：编辑/执行/提交
└─────────────┘
```

### 模式 7：Subagent Context Isolation（子代理上下文隔离）

**问题：**
```
长会话 → 上下文窗口积累一切
研究结果、规划讨论、代码编辑、测试输出、错误日志
→ 深度编辑时上下文被早期阶段无关材料污染
```

**解决：**
- 运行独立代理，各自上下文窗口、系统提示、限制工具访问
- 研究代理不能编辑代码
- 规划代理不能执行命令
- 每个子代理仅看到其特定任务所需内容

**何时使用：**
- ✅ 会话长、多阶段、涉及上下文需求非常不同的任务
- ⚠️ 权衡：协调开销（主代理决定传递什么给每个子代理、早期阶段重要细微差别可能在交接中丢失）

**视觉：**
```
┌─────────────────────────────────────────┐
│          Main Agent                     │
│  (协调、决策、交接)                      │
└─────────────────────────────────────────┘
        ↓ 交接
┌─────────────┬─────────────┬─────────────┐
│ Research    │ Planning    │ Editing     │
│ Agent       │ Agent       │ Agent       │
│ (只读)      │ (无执行)    │ (完全访问)   │
└─────────────┴─────────────┴─────────────┘
```

### 模式 8：Parallel Forking（并行的分支）

**问题：**
```
大任务可拆分为独立单元 → 仍顺序运行
20 文件迁移 → 20 顺序步骤
大多数文件互相无依赖
```

**解决：**
- 生成多个子代理并行
- 每个在隔离 git worktree 工作，仓库独立副本
- 父缓存上下文被每个 fork 重用，使并行分支 token 成本基本免费
- 所有分支完成时合并结果

**何时使用：**
- ✅ 任务分解为独立单元，互相无依赖
- ⚠️ 权衡：合并复杂性（并行分支触及重叠文件时合并产生比顺序工作更难解决的冲突）

**视觉：**
```
┌─────────────────────────┐
│     Parent Agent        │
│  (缓存上下文重用)        │
└─────────────────────────┘
        ↓ 并行 fork
┌───────┬───────┬───────┬───────┐
│Fork 1 │Fork 2 │Fork 3 │Fork 4 │
│Worktree│Worktree│Worktree│Worktree│
└───────┴───────┴───────┴───────┘
        ↓ 合并
┌─────────────────────────┐
│    Merged Result        │
└─────────────────────────┘
```

## 🛠️ 类别 3：Tools & Permissions（3 个模式）

### 模式 9：Progressive Tool Activation（渐进工具激活）

**问题：**
```
给代理访问所有可用工具 → 创建选择问题
60 工具可见 → 模型花更多时间决定用哪个
→ 更可能选错
```

**解决：**
- 从小默认集开始（Claude Code 少于 20 工具）
- 按需激活额外工具
- 默认：Read, Edit, Write, Bash, Grep, Glob 等 handful
- MCP 工具、远程工具、自定义技能仅需要时激活

**何时使用：**
- ✅ 代理访问多工具但大多数任务仅需少数
- ⚠️ 权衡：扩展逻辑增加复杂性（harness 必须决定何时激活、激活太晚意味着代理浪费轮次无正确能力）

**视觉：**
```
┌─────────────────────────┐
│  Default Tools (7)      │
│  Read, Edit, Write...   │
└─────────────────────────┘
        ↓ 按需激活
┌─────────────────────────┐
│  MCP Tools              │
│  Remote Tools           │
│  Custom Skills          │
└─────────────────────────┘
```

### 模式 10：Permission Gating（权限门控）

**问题：**
```
让代理运行任意 shell 命令无检查 → 危险
但要求用户批准每个命令 → 创建疲劳
→ 用户最终点击"yes"一切
```

**解决：**
- 执行前应用确定性预解析和每工具权限门控
- 每工具有独立 allow、ask、deny 规则，带模式匹配
- Shell 命令通过分类层解析动词、标志、目标评估风险
- auto-mode 分类器自动批准低风险行动
- 危险内容保留安全检查器

**何时使用：**
- ✅ 代理可执行 shell 命令或与外部系统交互
- ⚠️ 权衡：刚性（确定性分类器无法预期每个安全/危险命令、规则需持续调优）

**视觉：**
```
┌─────────────────────────┐
│   Shell Command         │
│   "rm -rf /tmp/cache"   │
└─────────────────────────┘
        ↓ 预解析
┌─────────────────────────┐
│   Classifier            │
│   verb: rm             │
│   flags: -rf           │
│   target: /tmp/cache   │
│   risk: LOW → AUTO     │
└─────────────────────────┘
        ↓ 执行
┌─────────────────────────┐
│   Execute (Auto)        │
└─────────────────────────┘
```

### 模式 11：Purpose-Built Tools（专用工具）

**问题：**
```
代理通过通用 shell 路由每个文件操作
(cat, sed, grep, find) → 命令更难审查
更难权限、更难模型正确使用
sed 编辑文件命令与腐蚀它结构相同
```

**解决：**
- 用专用工具替换通用 shell，每个常见操作：
  - FileReadTool
  - FileEditTool
  - GrepTool
  - GlobTool
- 每工具有类型化输入、约束范围、独立权限规则

**何时使用：**
- ✅ 代理频繁执行常见文件和搜索操作
- ⚠️ 权衡：灵活性（专用工具无法覆盖每个边缘情况、仍需通用 shell 作为回退）

**视觉：**
```
┌─────────────────────────────────────────┐
│   Purpose-Built Tools                   │
├─────────────────────────────────────────┤
│   FileReadTool   (类型化输入)           │
│   FileEditTool   (约束范围)             │
│   GrepTool       (独立权限)             │
│   GlobTool       (验证输入)             │
└─────────────────────────────────────────┘
        ↓ 对比
┌─────────────────────────────────────────┐
│   General Shell (cat/sed/grep/find)    │
│   难审查、难权限、难正确使用             │
└─────────────────────────────────────────┘
```

## ⚙️ 类别 4：Automation（1 个模式）

### 模式 12：Lifecycle Hooks（生命周期钩子）

**问题：**
```
某些行动必须每次发生无例外
(格式化器每次编辑后运行、命令执行前验证、工作目录变化时重载配置)
依靠模型通过提示指令记住这些 → 不可靠
→ 模型会忘记、跳过、重新解释指令取决于上下文压力
```

**解决：**
- 在代理生命周期特定点自动运行 shell 命令或其他行动
- 完全在提示外
- Claude Code 泄露代码包含 25+ 钩子点：
  - PreToolUse
  - PostToolUse
  - SessionStart
  - CwdChanged
  - 等等

**原则：**
> 任何必须每次发生的事情属于钩子，而非指令。

**何时使用：**
- ✅ 有不应跳过的不变行为
- ⚠️ 权衡：调试难度（钩子出错时比提示级指令更难诊断、钩子在对话外运行）

**视觉：**
```
┌─────────────────────────────────────────┐
│   Agent Lifecycle Hooks (25+)           │
├─────────────────────────────────────────┤
│   SessionStart    → 初始化              │
│   PreToolUse    → 验证                  │
│   PostToolUse   → 日志/巩固             │
│   CwdChanged    → 重载配置              │
│   SessionEnd    → 总结/清理             │
└─────────────────────────────────────────┘
```

## 💡 核心洞察

### 洞察 1：这些是基石级模式

**作者信念：**
> 这些模式不是临时技巧或产品特定功能。它们是代理 harness 设计的基石。

**持久相关的架构决策：**
- 记忆分层
- 上下文压缩
- 权限门控
- 生命周期钩子

**这些将保持相关当模型和工具在其下演进。**

---

### 洞察 2：Claude Code 泄露的价值

**罕见机会：**
- 看到这些模式在最前沿实现
- 生产代理被数十万开发者使用
- 可见性不会持久，但模式会

---

### 洞察 3：4 类别覆盖完整代理生命周期

| 类别 | 关注点 | 模式数 |
|------|--------|--------|
| **Memory & Context** | 代理知道什么 | 5 |
| **Workflow & Orchestration** | 代理如何工作 | 3 |
| **Tools & Permissions** | 代理被允许做什么 | 3 |
| **Automation** | 必须自动发生什么 | 1 |

## 📊 模式总结表

| # | 模式 | 类别 | 核心问题 | 解决 |
|---|------|------|----------|------|
| 1 | Project Instructions | Memory | 每次会话从零开始 | 持久项目级配置文件 |
| 2 | Scoped Instructions | Memory | 单文件太通用或太大 | 多范围动态加载 |
| 3 | Tiered Memory | Memory | 加载所有记忆浪费 token | 索引/主题/完整三层 |
| 4 | Memory Consolidation | Memory | 记忆随时间退化 | 后台垃圾回收 |
| 5 | Context Compaction | Memory | 命中上下文窗口限制 | 多阶段压缩 |
| 6 | Explore-Plan-Act | Workflow | 基于不完整理解编辑 | 三阶段递增权限 |
| 7 | Subagent Isolation | Workflow | 上下文被早期材料污染 | 独立代理独立上下文 |
| 8 | Parallel Forking | Workflow | 大任务顺序运行慢 | 并行 git worktree |
| 9 | Progressive Activation | Tools | 60 工具可见选择困难 | 小默认集按需激活 |
| 10 | Permission Gating | Tools | 任意 shell 危险或疲劳 | 预解析 + 分类 + 门控 |
| 11 | Purpose-Built Tools | Tools | 通用 shell 难审查 | 专用工具类型化输入 |
| 12 | Lifecycle Hooks | Automation | 模型忘记必须步骤 | 自动钩子在特定点 |
