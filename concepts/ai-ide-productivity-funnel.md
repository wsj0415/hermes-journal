## 编译知识

**AI IDE 生产力漏斗模型** 是 Google 团队提出的框架，用于解释为什么很多 AI 功能无法产生实际生产力影响。

### 漏斗层级（从上到下）

```
软件工程师行为
    ↓
[流失 1] 模型置信度不足，不生成建议
    ↓
[流失 2] 模型响应太慢或超时
    ↓
[流失 3] 建议质量差
    ↓
[流失 4] 用户没注意到建议
    ↓
[流失 5] 用户审查后拒绝
    ↓
实际应用 ML 建议 ← 只有这里产生生产力价值
```

### 核心洞察

**AI 建议必须同时满足 5 个条件才能产生价值:**

1. **生成** - 模型有足够置信度生成建议
2. **及时** - 低延迟响应（用户等待前完成）
3. **准确** - 建议质量高，符合用户意图
4. **可见** - 用户注意到建议的存在
5. **可用** - 用户审查后愿意接受

**关键结论:** 每个环节都会流失机会，必须系统性优化整个漏斗，而不是只关注模型质量。

## 两个 AI 功能案例

### Code Completion (代码完成)
- 实时预测用户下一步输入
- 基于 IDE 上下文
- 几个词到多行代码

### Transform Code (代码转换)
- 自然语言驱动的代码编辑
- 手动触发，用户选择范围 + 输入指令
- 适用于小型修改

## 方法论：数据驱动迭代

**Google 的做法:**
- 不依赖直觉或离线测试
- 所有决策基于在线实验数据
- A/B 测试 rollout 到内部开发者
- 持续测量生产力影响

**测量维度:**
- 建议接受率
- 编辑距离（用户修改多少）
- 任务完成时间
- 用户参与度

## 对 Agent 设计的启示

**将漏斗模型应用到 Hermes Agent:**

| 漏斗层级 | Agent 对应问题 | 优化方向 |
|----------|----------------|----------|
| 生成 | 任务理解准确率 | 改进系统提示词、上下文管理 |
| 及时 | 响应延迟 | 流式输出、并行工具调用 |
| 准确 | 输出质量 | RAG、技能库、自我验证 |
| 可见 | 进度可感知 | 流式消息、状态更新 |
| 可用 | 用户信任 | 可解释性、可撤销操作 |
