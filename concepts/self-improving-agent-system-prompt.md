## 编译知识

自改进代理系统提示词是一个雄心勃勃的代理操作系统，目标是构建一个从每个任务中学习、随时间推移不断进步的系统。核心工作流：**任务图 → 执行 → 验证 → 记忆更新 → 可见性 → 学习**。8 个核心能力：接受目标、转化为任务、路由执行、执行验证、保持记忆、学习改进、安全自主、自我改进。

**来源**：Tom Dörr (@tom_doerr) 推文推荐，fainir GitHub 项目

## 核心工作流

```
任务图 → 执行 → 验证 → 记忆更新 → 可见性 → 学习
```

### 6 个关键检查点

| 检查点 | 说明 |
|--------|------|
| **1. 任务图** | 将目标分解为明确的任务 |
| **2. 执行** | 路由到合适的代理或机器 |
| **3. 验证** | 确保工作正确完成 |
| **4. 记忆更新** | 保存有用的知识 |
| **5. 可见性** | 保持状态透明 |
| **6. 学习** | 从成功和失败中改进 |

### 执行期间的自我检查

- **如果偏离到纯聊天行为** → 停止并返回文件、任务、验证和实现
- **如果过早陷入复杂的多代理系统** → 简化，先做好单代理基线

## 北极星目标

构建一个持久的代理系统，能够：

| 能力 | 说明 |
|------|------|
| ✅ 接受目标 | 理解用户意图 |
| ✅ 转化为任务 | 将目标转为明确任务 |
| ✅ 路由执行 | 路由到有能力的代理或机器 |
| ✅ 执行验证 | 执行并验证工作 |
| ✅ 保持记忆 | 随时间保留记忆和知识 |
| ✅ 学习改进 | 从每次成功和失败中学习 |
| ✅ 安全自主 | 安全地增加自主性 |
| ✅ 自我改进 | 改进提示词、技能、工具、工作流、评估和架构 |
| ✅ 通用化 | 从狭窄演示扩展到通用计算机工作 |

---

## "最强大"的定义维度

不要仅将能力定义为基准分数或编码速度。跨以下维度定义：

| 维度 | 说明 |
|------|------|
| **广度** | 系统可执行的不同任务类型数量 |
| **深度** | 完成长、多步骤、模糊任务的能力 |
| **可靠性** | 正确完成（而非仅尝试）的能力 |
| **迁移** | 适应新领域和工具的能力 |
| **记忆** | 在数天、数项目和数机器间保存有用知识的能力 |
| **自改进** | 无需手动编辑每个行为即可改进的能力 |
| **治理** | 知道何时不行动、何时询问、何时升级的能力 |
| **经济性** | 在足够时选择更便宜方法、合理时选择更昂贵方法的能力 |
| **耐久性** | 在崩溃、重启、模型切换和运行时变化中生存的能力 |

## 成功指标

从一开始就追踪明确指标。至少追踪：

| 指标 | 说明 |
|------|------|
| **完成任务数** | 已完成的任务数量 |
| **验证完成数** | 经验证正确的任务数量 |
| **中位完成时间** | 完成任务的中位时间 |
| **单次成功成本** | 每次成功任务的成本 |
| **干预率** | 需要人工干预的比例 |
| **重试率** | 需要重试的比例 |
| **回归率** | 出现回归的比例 |
| **自主级别** | 按任务类型的自主级别 |
| **评估通过率** | 评估测试的通过率 |
| **重复运行稳定性** | 重复运行的稳定性 |
| **记忆复用率** | 记忆被复用的比例 |
| **主动 vs 被动** | 主动完成 vs 被动完成的比例 |
| **领域分布** | 按领域分布：编码/浏览器/文档/运营/研究/科学/商业 |

## 架构原则

### 运行时不可知，架构具体

**不要假设**：
- ❌ 特定产品
- ❌ 特定 IDE
- ❌ 特定 SDK
- ❌ 特定供应商

**要选择具体架构**：
- ✅ 明确的任务图
- ✅ 工作流和 harness
- ✅ 可见的会话
- ✅ 持久的记忆
- ✅ 控制平面状态
- ✅ 验证器层
- ✅ 工具和模型适配器
- ✅ 审批、预算和评估

**正确目标**：
- 一个通用用户代理界面
- 多个内部路由层（基于任务、技能、剧本、harness、模型、机器和验证器）

**不要混淆**：运行时可移植性 ≠ 架构模糊性

## 实现路径

### 两种有效默认实现路径

#### 路径 1：简单基线（第 1-2 周）

```
1. 单代理，无多代理复杂性
2. 基本任务图（待办事项文件）
3. 简单验证（运行测试、检查退出代码）
4. 文件基础记忆（markdown 文件）
5. 透明状态（单一状态文件）
6. 基本指标（完成任务数、成功率）
```

#### 路径 2：生产就绪（第 3-8 周）

```
1. 多代理协调（基于任务路由）
2. 复杂任务图（依赖、并行执行）
3. 多层验证（自动 + 人工审查）
4. 数据库记忆（SQLite + 向量搜索）
5. 完整状态追踪（事件日志）
6. 完整指标套件
```

---

## 关键洞察

1. **闭环优先** — 先证明完整循环，再扩展广度
2. **可观察 > 聪明** — 简单透明胜过复杂隐藏
3. **可靠性是设计出来的** — 在不可靠组件上构建可靠性
4. **自改进是必须的** — 系统必须能够改进自己
5. **指标驱动** — 从一开始就追踪明确指标
