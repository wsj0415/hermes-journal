# Agent Memory 架构演进

> 从第一性原理理解 Agent Memory 的设计演进：Python List → Markdown 文件 → 向量搜索 → 图向量混合 → 统一引擎
> 
> 来源：[[akshay-pachaar-agent-memory-2026]]

---

## 为什么 LLM 需要记忆

LLM 本质是无状态的。每次 API 调用都是全新的开始。ChatGPT 表现出的"记忆"是通过重新发送完整对话历史制造的幻觉。

这个技巧对 casual chat 有效，但构建真实 Agent 时会崩溃。

### 7 种失败模式

1. **上下文失忆**：Agent 询问已提供的信息
2. **零个性化**：每次交互都感觉通用
3. **多步任务失败**：中间状态在任务中静默丢失
4. **重复错误**：没有情景回忆，同样错误永远重复
5. **无知识积累**：每次会话从零开始
6. **幻觉来自缺口**：上下文溢出时，模型编造信息
7. **身份崩溃**：无连续性，无信任

### 为什么大上下文窗口不够

- **Lost in the Middle 效应**：相关信息在长上下文中间时，准确率下降 30%+
- **上下文是共享预算**：系统提示、检索文档、对话历史、输出都在争夺相同的 token
- **即使 100K token**，缺少持久化、优先级和显著性，原始上下文长度仍然不足

**记忆不是往 prompt 里塞更多文本，而是结构化 Agent 记住的内容，让它能找到重要的东西。**

## 认知科学框架

Lilian Weng 2023 年的公式成为默认框架：

```
Agent = LLM + Memory + Planning + Tool Use
```

四个同等重要的支柱。

### 人类记忆系统的映射

| 人类记忆 | 功能 | Agent 对应 |
|---------|------|-----------|
| 感觉记忆 | 捕捉原始感知输入，保持几分之一秒 | 输入感知层 |
| 工作记忆 | 主动思考发生的地方，~7±2 项 | 当前上下文窗口 |
| 长期记忆 | 无容量限制的持久存储 | 外部记忆系统 |

### 长期记忆的分类

- **情景记忆 (Episodic)**：特定事件（"周二 PostgreSQL 集群宕机"）
- **语义记忆 (Semantic)**：事实和概念（"PostgreSQL 是关系数据库"）
- **程序记忆 (Procedural)**：技能和工作流（"用户要求退款时，先检查购买日期"）

### 记忆巩固 (Memory Consolidation)

重复的特定事件蒸馏成通用知识的过程。Agent 如果注意到"用户一致偏好执行摘要"，应该将其转化为可复用规则。没有巩固，Agent 只是重播单个事件而非从中学习。

## 架构演进层次

### Layer 1: Python List

**方案**：将对话历史保存在列表中，每次调用重新发送。

```python
conversation = []
conversation.append({"role": "user", "content": "I have 4 apples"})
conversation.append({"role": "assistant", "content": "OK"})
# 下次调用时发送完整列表
```

**优点**：
- 多轮对话现在能工作
- 实现简单

**问题**：
- 列表无界增长，约 200 轮后命中上下文天花板
- 最早的消息静默丢失（用户名从第 1 轮消失）
- 无优先级，严格按时间顺序
- 全部在 RAM 中，进程结束记忆消失

---

### Layer 2: Markdown 文件持久化

**方案**：将记忆写入磁盘的 Markdown 文件。Claude Code 使用 `CLAUDE.md` 和 `MEMORY.md`。

```markdown
# User Facts
- Name: Sarah
- Company: Acme Corp
- Industry: FinTech
- Prefers: Executive summaries
```

**优点**：
- 持久化解决：重启脚本，对话仍在
- 人类可读，Git 友好
- 可手动编辑修正

**问题**：
- 3 个月后 2000+ 事实，500K+ tokens，超过上下文窗口
- 无法全部加载，需要选择性检索
- 只能关键词搜索，无法处理同义词、释义或跨事实连接
- **存储没有智能检索 = 没有目录的图书馆**

---

### Layer 3: 向量搜索

**方案**：嵌入 Markdown 块，按余弦相似度搜索。

```python
# "database" 匹配 "PostgreSQL" 因为向量在嵌入空间接近
# 同义词问题消失
```

**优点**：
- 语义理解：同义词、释义都能匹配
- 选择性检索：只加载相关片段

**新问题：关系推理缺失**

```
事实 1: Alice 是工程师
事实 2: 周二发生宕机
事实 3: Project Atlas 使用 PostgreSQL

用户问："Alice 的项目是否受周二宕机影响？"

→ 向量搜索会命中事实 1 和 2（提到 Alice 和周二）
→ 但关键桥梁"Project Atlas 使用 PostgreSQL"不会 surfaced
→ 每个事实是嵌入空间中的孤立点，连接结构对向量不可见
```

**这不是边缘情况，是真实世界问题的常态**：
- 人属于团队
- 团队拥有项目
- 项目依赖系统
- 系统发生事故

任何跨越 2+ 跳的问题超出扁平向量检索的能力。

## Cognee: 三存储一引擎

Cognee 是开源知识引擎，为 Agent Memory 构建。结合向量搜索 + 知识图谱 + 关系证明层。

### 四调用 API

```python
await cognee.add(data)      # 添加数据
await cognee.cognify()      # 处理成结构化知识
await cognee.search(query)  # 检索
await cognee.memify()       # 优化记忆结构
```

### 三存储架构

| 存储 | 捕获维度 | 技术栈 |
|------|----------|--------|
| 关系存储 | **证明**：数据来源、时间、访问权限 | SQLite / PostgreSQL |
| 向量存储 | **语义**：内容含义、相似度 | LanceDB / Qdrant / pgvector |
| 图存储 | **关系**：实体连接、因果关系 | Kuzu / Neo4j / FalkorDB |

**为什么需要三个存储**：扁平化任何一个都会丢失对检索准确性重要的信息。

**默认栈**：SQLite + LanceDB + Kuzu，完全嵌入式，基于文件。
```bash
pip install cognee + LLM API key = 几分钟内运行
```

无需 Docker，无需外部服务。生产环境可替换为 PostgreSQL + Qdrant + Neo4j，API 不变。

---

### Cognify 管道

`cognee.cognify()` 运行多阶段管道：

1. 文档分类（类型和领域）
2. 多租户访问控制的权限检查
3. 块提取（尊重大段落结构，非固定大小切割）
4. LLM 实体和关系提取，内容哈希自动去重
5. 摘要生成用于高效检索
6. 双重索引到向量存储（嵌入）和图存储（边）

**去重步骤关键**：同一实体出现在 50 个文档中，Cognee 合并为单个图节点 + 50 个入边。Agent 不再看到 50 个不同的"Alice"。

**增量处理**：只有新/更新的文件被重新处理。

**双重表示核心技巧**：
- 每个图节点有对应嵌入
- 通过向量进入（找到语义相似内容）
- 通过图退出（追踪连接到相关实体）
- 或反向操作
- 这使得多跳查询工作而不牺牲语义搜索

---

### Memify: 学习进化的记忆

`memify()` 在图上运行 RL 启发的优化：

- **加强有用路径**：导致良好检索的路径
- **修剪陈旧节点**：未被触碰的节点
- **自动调整边权重**：基于实际使用
- **添加衍生事实**：识别隐含关系

**示例**：客服 Agent 的图自然加强通过产品文档和退款政策的路径，同时让很少查询的 HR 边衰减。图随时间发展出自己的相关性感知。

## 实战：用 Cognee 构建 Agent

完整模式将 Cognee 接入 perceive-think-act 循环：

```
记忆循环：ingest → extract → store → retrieve → respond → store again
```

每次循环丰富知识图，增量处理意味着只为新内容付费索引。

## 决策框架

**关键问题**："我的 Agent 需要记住什么？它会回答什么问题？"

| 查询类型 | 推荐架构 |
|----------|----------|
| 仅需相似度搜索（"找到像这样的对话"） | 纯向量记忆 |
| 跨实体边界（"Alice 的项目是否受周二宕机影响？"） | 图遍历必需 |

**自己组装的代价**：
- 粘合向量数据库 + 图数据库 + 关系存储 + 实体提取器 + 去重管道 + 边权重系统
- 数周的基础设施工作，还没写一行 Agent 逻辑

**Cognee 优势**：
- 四个 API 调用完成
- 嵌入式默认几分钟运行
- 可替换后端支持生产，无需改 Agent 代码

## 核心洞察

**智能需要结构，不只是存储。**

三种存储范式（关系、向量、图）不是竞争选项，是同一记忆系统的互补层。这样处理才能将无状态 LLM 包装变成真正学习的东西。

**起点问题**："你的 Agent 明天需要记住什么，但今天忘记了？"从那里开始。
