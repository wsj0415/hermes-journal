## 编译知识

Agentic Harness 是构建代理应用程序时的 12 个可复用设计模式，来自对 Claude Code 泄露代码的分析。分为 4 个类别：记忆与上下文（5 个）、工作流与编排（3 个）、工具与权限（3 个）、自动化（1 个）。

**核心价值**：这些模式代表了代理设计的基石级架构决策，可直接应用于 Hermes Agent、OpenClaw 等系统。

## 类别 1：Memory & Context（5 个模式）

### 模式 1：Project Instructions（项目指令）

**问题**：
```
没有持久指令文件 → 每次会话从零开始
→ 用户重复相同约定
→ 代理在第 5 次会话犯第 1 次的错误
```

**解决**：
- 引入持久项目级配置文件（`CLAUDE.md` / `AGENTS.md`）
- 每次会话开始自动加载
- 定义：构建命令、测试命令、架构规则、命名约定、编码标准
- 随仓库一起分发，而非用户剪贴板

**何时使用**：
- ✅ 代理跨多个会话在代码库上工作
- ⚠️ 权衡：维护负担（文件必须随项目演进而更新）

**视觉**：
```
┌─────────────────────────┐
│   CLAUDE.md / AGENTS.md │
│   - 构建命令             │
│   - 测试命令             │
│   - 架构规则             │
│   - 命名约定             │
│   - 编码标准             │
└─────────────────────────┘
        ↓ 自动加载
┌─────────────────────────┐
│     Agent Session       │
└─────────────────────────┘
```

---

### 模式 2：Scoped Instructions（范围指令）

**问题**：
```
单指令文件 → 小项目可行
大代码库 → 要么变成被忽视的巨大 blob
         要么保持太通用对任何目录都无用
```

**解决**：
- 从多个文件动态加载指令，不同范围：
  - 组织级
  - 用户级
  - 项目根目录
  - 父目录
  - 子目录
- 代理看到不同规则取决于在代码库何处工作
- Import 语法允许跨文件拆分大指令集无重复

**何时使用**：
- ✅ Monorepo、多语言项目、不同目录遵循不同约定
- ⚠️ 权衡：可发现性（指令分布在多文件中难理解）

---

### 模式 3：Session Memory（会话记忆）

**问题**：
```
会话间无记忆 → 用户重复背景信息
             → 代理忘记之前的决策
             → 进展丢失
```

**解决**：
- 每个会话结束写入记忆文件
- 下次会话开始读取
- 捕获：用户偏好、项目决策、进行中的工作、已尝试的方法

**何时使用**：
- ✅ 多会话项目、迭代开发
- ⚠️ 权衡：记忆膨胀（需要定期清理）

---

### 模式 4：Context Pruning（上下文修剪）

**问题**：
```
长会话 → 上下文窗口填满
       → 早期重要信息被丢弃
       → 代理失去关键背景
```

**解决**：
- 主动监控上下文使用
- 基于相关性评分删除旧消息
- 保留：用户偏好、关键决策、当前任务
- 删除：中间步骤、已解决的讨论、临时想法

**何时使用**：
- ✅ 长会话、复杂项目
- ⚠️ 权衡：可能丢失看似无关但后来重要的信息

---

### 模式 5：Cross-Session Learning（跨会话学习）

**问题**：
```
会话 1：捕获定价测试显示 40% 流失率
会话 2：捕获竞争对手取消了免费层级
会话 3：捕获 onboarding 用户流失较少
三个独立的观察，Claude 从未将它们连接起来
```

**解决**：
- 知识层次结构：
  ```
  knowledge/
  ├── observations/        # 原始观察
  ├── hypotheses/          # 需要更多数据的假设
  └── rules/               # 确认规则
  ```
- 提升循环：
  ```
  观察 → 假设 → (多次确认) → 规则
                           ↓
                      (新数据矛盾)
                           ↓
                      降级回假设
  ```

**何时使用**：
- ✅ 长期项目、需要累积洞察
- ⚠️ 权衡：复杂性（需要主动维护知识结构）

## 类别 2：Workflow & Orchestration（3 个模式）

### 模式 6：Multi-Agent Handoff（多代理交接）

**问题**：
```
单代理处理所有任务 → 上下文切换开销
                   → 专业知识和通用知识混合
                   → 效率低下
```

**解决**：
- 专用代理用于不同任务（研究、编码、审查）
- 清晰交接协议
- 共享上下文格式

**何时使用**：
- ✅ 复杂项目需要多种专业知识
- ⚠️ 权衡：协调开销

---

### 模式 7：Progressive Disclosure（逐步披露）

**问题**：
```
一次性加载整个代码库 → 上下文 overwhelmed
                     → 代理错过重要细节
```

**解决**：
- 从高层摘要开始
- 根据任务需要深入具体文件
- 按需加载上下文

**何时使用**：
- ✅ 大代码库、探索性任务
- ⚠️ 权衡：可能错过跨文件模式

---

### 模式 8：Checkpoint & Resume（检查点与恢复）

**问题**：
```
长任务被中断 → 进展丢失
             → 下次会话重新做相同工作
```

**解决**：
- 定期写入检查点文件
- 捕获：完成的工作、下一步、当前状态
- 从检查点恢复而非重新开始

**何时使用**：
- ✅ 多日项目、可能中断的任务
- ⚠️ 权衡：存储开销

---

## 类别 3：Tools & Permissions（3 个模式）

### 模式 9：Tool Discovery（工具发现）

**问题**：
```
代理不知道可用工具 → 无法使用它们
                   → 用户必须手动告诉
```

**解决**：
- 启动时自动发现工具
- 工具描述包含能力、限制、使用示例
- 动态注册新工具

**何时使用**：
- ✅ 工具集频繁变化
- ⚠️ 权衡：启动时间

---

### 模式 10：Permission Gates（权限门）

**问题**：
```
代理可执行任何操作 → 危险操作风险
                   → 用户不信任
```

**解决**：
- 定义需要批准的操作（删除、部署、API 调用）
- 自动执行安全操作
- 清晰提示危险操作

**何时使用**：
- ✅ 生产环境、敏感操作
- ⚠️ 权衡：用户体验摩擦

---

### 模式 11：Tool Chaining（工具链）

**问题**：
```
单工具调用 → 复杂任务需要多步骤
           → 用户必须手动协调
```

**解决**：
- 预定义工具链用于常见工作流
- 错误处理和重试逻辑
- 中间结果缓存

**何时使用**：
- ✅ 重复性多步骤任务
- ⚠️ 权衡：灵活性降低

---

## 类别 4：Automation（1 个模式）

### 模式 12：Scheduled Tasks（定时任务）

**问题**：
```
手动触发任务 → 容易忘记
             → 不一致执行
```

**解决**：
- 定义定时任务（每日摘要、每周报告、定期备份）
- 自动执行和交付
- 失败通知

**何时使用**：
- ✅ 定期报告、监控、维护任务
- ⚠️ 权衡：调度复杂性

## 应用建议

### 立即实施（高优先级）

| 模式 | 实施难度 | 影响力 |
|------|----------|--------|
| Project Instructions | 低 | 高 |
| Session Memory | 低 | 高 |
| Permission Gates | 中 | 高 |

### 中期实施（中优先级）

| 模式 | 实施难度 | 影响力 |
|------|----------|--------|
| Scoped Instructions | 中 | 中 |
| Context Pruning | 中 | 中 |
| Checkpoint & Resume | 中 | 中 |

### 长期实施（低优先级）

| 模式 | 实施难度 | 影响力 |
|------|----------|--------|
| Cross-Session Learning | 高 | 高 |
| Multi-Agent Handoff | 高 | 中 |
| Tool Chaining | 高 | 中 |
